## 2. 步骤一：激活 GitHub Pages（Source 设置为 GitHub Actions）✅ 已完成

### 2.1 为什么需要？

项目已配置 `.github/workflows/deploy-knowledge-graph.yml`，用于在每次推送到 `main` 分支时自动构建并部署知识图谱站点到 GitHub Pages。然而，GitHub 默认不会自动识别并使用 Actions 作为 Pages 的部署源，必须手动切换。

### 2.2 操作路径

1. 打开仓库主页：`https://github.com/<org>/AnalysisDataFlow`
2. 点击顶部菜单栏 **Settings**（设置）
3. 在左侧边栏中，找到 **Pages**（位于 Code and automation 分类下）
4. 在 **Build and deployment** 区域：
   - **Source**: 从 `Deploy from a branch` 切换为 **`GitHub Actions`**

### 2.3 预期结果

- 页面显示："Your site is ready to be published at `https://<org>.github.io/AnalysisDataFlow/`"
- 工作流 `deploy-knowledge-graph.yml` 在下次 push 到 main 时会自动触发并成功部署
- 若工作流之前因 Source 未设置而失败，切换后重新触发即可修复

### 2.4 验证方法

1. 在 Settings > Pages 页面确认 Source 显示为 "GitHub Actions"
2. 在仓库首页点击 **Actions** 标签
3. 找到 "Deploy Knowledge Graph" 工作流，手动点击 **Run workflow** 触发一次
4. 等待约 2-3 分钟，确认工作流状态为 ✅ 绿色通过
5. 访问 `https://<org>.github.io/AnalysisDataFlow/` 验证站点可正常加载

### 2.5 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Actions 工作流找不到 "Pages" 部署环境 | Source 未设置为 GitHub Actions | 重新检查 Settings > Pages |
| 部署成功但页面 404 | 仓库为私有 / CNAME 未生效 | 确认仓库已 Public，或检查 `CNAME` 文件 |
| 自定义域名无法访问 | DNS 记录未指向 GitHub Pages | 检查域名 DNS 的 A/AAAA/CNAME 记录 |

## 3. 步骤二：申请 Algolia DocSearch

### 3.1 为什么需要？

项目已在前端代码中集成 Algolia DocSearch 的搜索组件和本地搜索 fallback。但要使 Algolia 爬虫定期索引站点内容，必须向 Algolia 提交 DocSearch 申请。该申请免费，但审核周期通常为 **3-5 个工作日**。

### 3.2 申请入口

打开 Algolia DocSearch 申请页面：

```
https://docsearch.algolia.com/apply/
```

### 3.3 表单填写指南

| 字段 | 建议填写内容 |
|------|-------------|
| **Email** | 项目管理员或维护者的邮箱（需能接收后续配置邮件） |
| **Repository URL** | `https://github.com/<org>/AnalysisDataFlow` |
| **Website URL** | `https://<org>.github.io/AnalysisDataFlow/`（GitHub Pages 地址）或自定义域名 |
| **Project Name** | `AnalysisDataFlow` |
| **Description** | "A comprehensive knowledge base for stream processing theory, formal methods, and Apache Flink ecosystem." |
| **Is your project open source?** | ✅ Yes |
| **License** | 根据项目实际情况填写（如 CC BY-SA 4.0 / Apache 2.0） |

### 3.4 申请后流程

1. **提交申请** → Algolia 团队会在 3-5 个工作日内审核
2. **收到欢迎邮件** → 邮件中包含 `appId`、`apiKey`、`indexName` 和配置脚本
3. **更新项目配置** → 将收到的 `appId` 和 `apiKey` 更新到以下位置：
   - 前端搜索组件初始化代码（如 `KNOWLEDGE-GRAPH/search.js` 或对应位置）
   - `.github/workflows/deploy-knowledge-graph.yml` 中的环境变量（如有）
4. **触发重新部署** → Algolia 爬虫会自动开始索引站点内容

### 3.5 验证方法

1. 收到 Algolia 邮件后，登录 [Algolia Dashboard](https://www.algolia.com/apps)
2. 找到对应的 DocSearch 应用和索引
3. 在 **Search** 标签下尝试搜索几个关键词（如 "Flink Checkpoint"、"Watermark"）
4. 在项目站点前台使用搜索框，确认能返回 Algolia 的结果

### 3.6 邮件模板（可选）

如需向 Algolia 团队补充说明，可使用以下邮件内容：

```text
Subject: DocSearch Application for AnalysisDataFlow

Hi Algolia DocSearch Team,

We have applied for DocSearch for our open-source knowledge base project:
- Repository: https://github.com/<org>/AnalysisDataFlow
- Site: https://<org>.github.io/AnalysisDataFlow/

The site is built with static Markdown files and follows the standard
DocSearch sitemap structure. We have already integrated the DocSearch
frontend component and are ready to receive the crawler configuration.

Please let us know if any additional information is needed.

Best regards,
[Your Name]
```

## 4. 步骤三：开启 GitHub Discussions ✅ 已完成

### 4.1 为什么需要？

项目已配置 `.github/DISCUSSION_TEMPLATE/` 讨论模板和 `discussions/` 目录内容。GitHub Discussions 是社区成员进行问答、分享想法和展示成果的理想场所。但该功能默认关闭，需要管理员手动启用。

### 4.2 操作路径

1. 打开仓库主页：`https://github.com/<org>/AnalysisDataFlow`
2. 点击顶部菜单栏 **Settings**
3. 在左侧边栏最下方找到 **General**
4. 向下滚动到 **Features** 区域
5. 勾选 **Discussions** 复选框

### 4.3 预期结果

- 仓库顶部导航栏出现 **Discussions** 标签
- 用户可以在 Discussions 中创建新的讨论，并自动看到预设的模板分类（Q&A、Ideas、Show and tell）

### 4.4 后续配置建议

启用 Discussions 后，建议完成以下配置以增强社区体验：

1. **设置默认讨论分类**：
   - 进入 **Discussions** → **Categories**
   - 确认已有：General、Q&A、Ideas、Show and tell
   - 可根据需要新增 "Translation"、"Bug Report" 等分类

2. **置顶欢迎帖**：
   - 创建一篇置顶讨论帖（Pinned discussion）
   - 内容可引用 `COMMUNITY/discussion-topics.md` 中的引导语

3. **链接到社区指南**：
   - 在仓库 `README.md` 或 `COMMUNITY/README.md` 中添加 Discussions 入口链接

### 4.5 验证方法

1. 以非管理员账号（或隐身窗口）访问仓库主页
2. 确认顶部导航栏可见 **Discussions**
3. 点击 "New discussion"，确认模板正常加载
4. 发布一篇测试讨论，确认流程顺畅
