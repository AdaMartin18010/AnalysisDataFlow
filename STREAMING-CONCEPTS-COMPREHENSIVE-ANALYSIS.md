### 3.2 关键概念精确定义

#### 3.2.1 流的数学定义

**项目当前定义** (Def-S-04-03):

```
S = (M, μ, ⪯, t_e, t_p)
```

**建议完善** (对齐权威论文):

```
S = (M, μ, ⪯, ∥, t_e, t_p, λ)
```

新增:

- ∥ ⊆ M × M: 并发关系 (并发事件不可比较)
- λ: M → L: 标签函数 (支持有色Petri网)

#### 3.2.2 Watermark的完整语义

**当前定义** (Def-S-04-04):

```
w: Stream → T ∪ {+∞}
```

**建议增强**:

```
Watermark(w, L, δ) = ∀r ∈ S.
  - t_e(r) ≤ w: 已经到达
  - w < t_e(r) ≤ w + L: 可能迟到 (概率 δ)
  - t_e(r) > w + L: 永远不会到达
```

#### 3.2.3 Exactly-Once端到端语义

**当前定义** (Def-S-08-04):

```
Exactly-Once ≡ ∀e ∈ Input. effect(e) = 1
```

**建议增强** (三要素分解):

```
Exactly-Once ≡ Source(replayable) ∧ Process(idempotent) ∧ Sink(atomic)
```

## 八、风险与应对

### 8.1 风险识别

| 风险ID | 风险描述 | 概率 | 影响 | 等级 |
|--------|----------|------|------|------|
| R1 | 形式化证明复杂度超预期 | 中 | 高 | 高 |
| R2 | 权威来源更新导致对齐工作变化 | 低 | 中 | 低 |
| R3 | 资源不足导致进度延迟 | 中 | 中 | 中 |
| R4 | 新发现的概念缺口增加工作量 | 中 | 高 | 高 |

### 8.2 应对措施

**R1: 形式化证明复杂度超预期**

- 应对: 引入Smart Casual验证方法，先建立工程论证再追求严格证明
- 缓解: 分阶段交付，先完成框架再完善细节

**R2: 权威来源更新**

- 应对: 建立版本跟踪机制，定期同步权威文档变更
- 缓解: 关注FLIP、RFC等变更预告

**R3: 资源不足**

- 应对: 优先级排序，确保P0任务完成
- 缓解: 任务并行化，利用已有推导链复用

**R4: 新发现概念缺口**

- 应对: 建立"已知缺口"清单，区分Must-have和Nice-to-have
- 缓解: 敏捷调整计划，及时沟通范围变更
