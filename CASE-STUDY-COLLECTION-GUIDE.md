# 🏭 流计算工业案例征集指南

> 建立真实生产环境验证案例的收集机制

---

## 📢 征集说明

### 征集目的

本项目致力于构建流计算领域最全面、最权威的工业实践知识库。通过收集来自真实生产环境的案例，我们希望：

- **验证理论** - 将学术理论与工业实践相结合
- **分享经验** - 让成功案例惠及更广泛的开发者
- **建立标准** - 形成行业公认的最佳实践
- **促进交流** - 搭建工业界与学术界的桥梁

### 征集对象

我们欢迎以下类型的案例提交：

| 类型 | 说明 | 示例 |
|------|------|------|
| 架构设计案例 | 系统整体架构与技术选型 | 基于 Flink 的实时数仓架构 |
| 性能优化案例 | 吞吐量、延迟优化实践 | 将延迟从秒级优化到毫秒级 |
| 故障处理案例 | 生产环境问题排查与解决 | Checkpoint 超时问题诊断 |
| 业务应用案例 | 具体业务场景的流计算应用 | 实时风控、推荐系统 |
| 迁移改造案例 | 从其他系统迁移到流计算 | Storm 迁移到 Flink |

### 案例要求

- ✅ 来自真实生产环境（非 POC/测试环境）
- ✅ 包含量化的性能指标和数据规模
- ✅ 提供系统架构图或技术方案
- ✅ 分享有实际价值的经验教训
- ✅ 不包含敏感信息和商业机密

## 📝 案例模板

### 提交方式

#### 方式一：GitHub Issue（推荐）

使用专门的 Issue 模板：

1. 访问 [GitHub Issues]
2. 选择 **🏭 工业案例征集** 模板
3. 按照模板指引填写完整信息
4. 提交 Issue

模板位置：`.github/ISSUE_TEMPLATE/production_case_study.md`

#### 方式二：直接提交文件

1. Fork 本仓库
2. 复制 [case-studies/template.md](./case-studies/template.md)
3. 在 `case-studies/submitted/` 目录下创建新文件
4. 提交 Pull Request

### 模板结构

完整的案例应包含以下八个部分：

```
1. 基本信息        - 公司、行业、联系人
2. 使用场景        - 业务背景、核心价值
3. 数据规模        - 吞吐量、集群规模、延迟要求
4. 架构图          - 系统架构、技术栈、数据流
5. 经验总结        - 技术选型、挑战与解决方案、最佳实践
6. 项目成果        - 量化指标、业务价值
7. 隐私与授权      - 敏感信息处理、授权声明
8. 附加材料        - 文档链接、代码片段
```

## 🔒 隐私保护说明

### 数据安全承诺

我们深知生产环境信息的敏感性，因此承诺：

1. **分级匿名** - 支持完全公开、半匿名、完全匿名三种级别
2. **严格审核** - 所有案例发布前经过敏感信息审查
3. **授权控制** - 未经明确授权，不会公开任何公司信息
4. **随时撤销** - 可随时申请更新或撤下已发布内容

### 脱敏指南

提交案例前，请确保以下内容已脱敏：

| 敏感类型 | 处理方式 | 示例 |
|----------|----------|------|
| 内部主机名/IP | 使用占位符 | `flink-master.internal` → `[jobmanager]` |
| 项目代号 | 使用描述性名称 | `Project Phoenix` → `实时风控系统` |
| 业务绝对数值 | 使用相对值/百分比 | `日交易额 10亿` → `日交易额增长 30%` |
| 内部 API | 使用简化描述 | 移除具体接口路径 |
| 员工信息 | 使用昵称/匿名 | 仅保留技术贡献者 GitHub ID |

### 授权级别

提交者可选择以下公开级别：

- **🔓 完全公开**
  - 可公开公司名称、Logo
  - 可公开所有技术细节和架构图
  - 可在技术大会等场合分享

- **🔐 半匿名**
  - 公开行业类型（如：某电商平台）
  - 公开技术方案和架构
  - 隐藏公司具体名称

- **🔒 完全匿名**
  - 仅展示技术架构和解决方案
  - 不透露任何公司信息
  - 适用于内部信息管控严格的公司

## ✅ 审核流程

### 流程概览

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  提交   │ → │ 格式审核 │ → │ 技术审核 │ → │ 隐私审核 │ → │ 收录发布 │
│ (Day 0) │    │(Day 1-2)│    │(Day 3-5)│    │(Day 6-7)│    │(Day 8+) │
└─────────┘    └────┬────┘    └────┬────┘    └────┬────┘    └─────────┘
                    │              │              │
                    ↓              ↓              ↓
              ┌─────────┐    ┌─────────┐    ┌─────────┐
              │ 反馈修改 │    │ 专家验证 │    │ 最终确认 │
              │(如需要) │    │(如需要) │    │(如需要) │
              └─────────┘    └─────────┘    └─────────┘
```

### 各阶段说明

#### 1️⃣ 格式审核（1-2个工作日）

- 检查模板字段完整性
- 验证必填项是否已填写
- 确认架构图和数据规模已提供
- 如发现问题，通过 Issue 评论反馈

#### 2️⃣ 技术审核（3-5个工作日）

- 技术专家验证架构合理性
- 确认性能数据的可信度
- 评估技术方案的最佳实践价值
- 可能需要补充技术细节

#### 3️⃣ 隐私审核（1-2个工作日）

- 检查敏感信息脱敏情况
- 确认授权声明完整
- 验证公开级别设置
- 必要时进行模糊处理

#### 4️⃣ 收录发布

- 案例移动到 `case-studies/verified/` 目录
- 按行业分类归档
- 更新案例库首页统计
- 通知提交者并致谢

### 审核标准

| 维度 | 权重 | 说明 |
|------|------|------|
| 完整性 | 25% | 模板字段填写完整 |
| 技术价值 | 30% | 方案有创新性和参考价值 |
| 数据真实性 | 25% | 提供可验证的性能数据 |
| 隐私合规 | 20% | 无敏感信息泄露风险 |
