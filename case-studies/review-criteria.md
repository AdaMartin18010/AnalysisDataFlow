# ✅ 案例初审评分标准

> 工业案例征集的审核标准与评分细则

---

## 审核流程概览

```
提交 → 格式审核 → 技术审核 → 隐私审核 → 收录发布
       (1-2天)     (3-5天)     (1-2天)
         ↑           ↑           ↑
      反馈修改    专家验证    最终确认
```

---

## 评分维度与权重

| 维度 | 权重 | 说明 |
|------|------|------|
| **完整性** | 25% | 模板字段填写完整 |
| **技术价值** | 30% | 方案有创新性和参考价值 |
| **数据真实性** | 25% | 提供可验证的性能数据 |
| **隐私合规** | 20% | 无敏感信息泄露风险 |

**通过分数线**：总分 ≥ 70 分，且各维度单项 ≥ 50 分

---

## 1. 完整性评分（25%）

评估案例是否完整填写了模板要求的各个部分。

### 1.1 基本信息（5分）

| 检查项 | 分值 | 要求 |
|--------|------|------|
| 公司名称/匿名标识 | 1分 | 明确标识 |
| 行业类型 | 1分 | 明确所属行业 |
| 技术栈 | 1分 | 列出主要技术 |
| 联系人信息 | 1分 | 提供联系邮箱或 GitHub ID |
| 授权声明 | 1分 | 明确公开级别 |

**评分标准**：

- 5分：所有信息完整且准确
- 3-4分：缺少1-2项次要信息
- 1-2分：缺少关键信息
- 0分：基本信息严重缺失

### 1.2 使用场景（5分）

| 检查项 | 分值 | 要求 |
|--------|------|------|
| 业务背景 | 2分 | 清晰描述业务场景 |
| 问题描述 | 2分 | 明确要解决什么问题 |
| 核心价值 | 1分 | 说明带来的业务价值 |

**评分标准**：

- 5分：场景描述清晰完整，逻辑通顺
- 3-4分：基本完整但部分描述不够详细
- 1-2分：描述模糊或缺少关键信息
- 0分：场景描述缺失或无法理解

### 1.3 数据规模（5分）

| 检查项 | 分值 | 要求 |
|--------|------|------|
| 吞吐量（QPS/TPS） | 2分 | 提供具体数值 |
| 数据量（GB/TB/PB） | 2分 | 提供数据规模 |
| 延迟要求 | 1分 | 说明延迟 SLA |

**评分标准**：

- 5分：提供量化数据，有具体数值
- 3-4分：提供相对描述（如"高并发"）
- 1-2分：数据规模描述模糊
- 0分：完全未提及数据规模

### 1.4 架构图（5分）

| 检查项 | 分值 | 要求 |
|--------|------|------|
| 系统架构图 | 3分 | 提供架构图（图片或文字描述） |
| 数据流说明 | 1分 | 描述数据流向 |
| 组件说明 | 1分 | 说明各组件职责 |

**评分标准**：

- 5分：架构图清晰，组件和数据流完整
- 3-4分：有架构图但缺少部分说明
- 1-2分：架构描述过于简单
- 0分：无架构图或架构描述

### 1.5 经验总结（5分）

| 检查项 | 分值 | 要求 |
|--------|------|------|
| 技术选型说明 | 2分 | 为什么选择该技术方案 |
| 遇到的挑战 | 2分 | 描述遇到的问题 |
| 解决方案 | 1分 | 说明如何解决 |

**评分标准**：

- 5分：经验总结深入，有可复用的最佳实践
- 3-4分：有经验总结但深度不够
- 1-2分：经验描述过于简单
- 0分：无经验总结

---

## 2. 技术价值评分（30%）

评估案例的技术创新性和对社区的参考价值。

### 2.1 技术方案创新性（10分）

| 等级 | 分值 | 说明 |
|------|------|------|
| 优秀 | 9-10分 | 有独特的技术创新或架构设计 |
| 良好 | 7-8分 | 在常规方案基础上有优化改进 |
| 一般 | 5-6分 | 采用业界常用方案 |
| 较弱 | 3-4分 | 技术方案较为常规 |
| 差 | 0-2分 | 技术方案存在问题或过于简单 |

**评估要点**：

- 是否解决了复杂的技术问题
- 是否有独特的架构设计
- 是否采用了创新的技术手段
- 是否有性能或效率的显著提升

### 2.2 参考价值（10分）

| 等级 | 分值 | 说明 |
|------|------|------|
| 优秀 | 9-10分 | 对社区有重要参考价值，可广泛复用 |
| 良好 | 7-8分 | 对特定场景有较高参考价值 |
| 一般 | 5-6分 | 有一定参考价值 |
| 较弱 | 3-4分 | 参考价值有限 |
| 差 | 0-2分 | 难以复用或参考 |

**评估要点**：

- 技术方案是否具有通用性
- 经验总结是否可复用
- 是否提供了足够的实现细节
- 对类似场景是否有指导意义

### 2.3 技术深度（10分）

| 等级 | 分值 | 说明 |
|------|------|------|
| 优秀 | 9-10分 | 深入剖析技术原理和实现细节 |
| 良好 | 7-8分 | 有技术细节但部分可更深入 |
| 一般 | 5-6分 | 有基本的技术描述 |
| 较弱 | 3-4分 | 技术描述较为浅显 |
| 差 | 0-2分 | 缺乏技术细节 |

**评估要点**：

- 是否深入解释技术原理
- 是否提供了配置参数和调优细节
- 是否有性能瓶颈分析和优化过程
- 是否有代码片段或配置示例

---

## 3. 数据真实性评分（25%）

评估案例中的性能数据是否真实可信。

### 3.1 数据完整性（10分）

| 检查项 | 分值 | 要求 |
|--------|------|------|
| 吞吐量数据 | 3分 | 提供 QPS/TPS 等数据 |
| 延迟数据 | 3分 | 提供延迟分布（p50/p99等） |
| 资源使用 | 2分 | 提供 CPU/内存/存储等信息 |
| 可用性数据 | 2分 | 提供 SLA/可用性指标 |

**评分标准**：

- 10分：所有关键数据完整
- 7-9分：缺少1项次要数据
- 4-6分：缺少2项数据
- 1-3分：仅提供部分数据
- 0分：无性能数据

### 3.2 数据可信度（10分）

| 等级 | 分值 | 说明 |
|------|------|------|
| 优秀 | 9-10分 | 数据来源清晰，有监控截图或日志佐证 |
| 良好 | 7-8分 | 数据合理，有文字说明来源 |
| 一般 | 5-6分 | 数据基本合理 |
| 可疑 | 3-4分 | 数据存在疑问 |
| 不可信 | 0-2分 | 数据明显不合理或无法验证 |

**评估要点**：

- 数据是否符合该规模系统的常见指标
- 是否说明了数据的测量方式和时间范围
- 数据之间是否逻辑一致
- 是否提供了数据验证的方式

### 3.3 数据粒度（5分）

| 等级 | 分值 | 说明 |
|------|------|------|
| 优秀 | 5分 | 提供分位值（p50/p95/p99） |
| 良好 | 4分 | 提供平均值和峰值 |
| 一般 | 3分 | 提供平均值 |
| 较弱 | 1-2分 | 仅提供粗略范围 |
| 差 | 0分 | 无具体数据 |

---

## 4. 隐私合规评分（20%）

评估案例是否符合隐私保护要求。

### 4.1 敏感信息检查（10分）

| 检查项 | 分值 | 扣分规则 |
|--------|------|----------|
| 内部主机名/IP | 2分 | 发现真实内部地址扣2分 |
| 内部项目代号 | 2分 | 发现敏感项目名扣2分 |
| 业务绝对数值 | 2分 | 发现敏感业务数据扣2分 |
| 内部 API/接口 | 2分 | 发现内部接口信息扣2分 |
| 员工个人信息 | 2分 | 发现敏感员工信息扣2分 |

**评分标准**：

- 10分：无敏感信息泄露
- 5-9分：存在轻微敏感信息，已脱敏处理
- 1-4分：存在较多敏感信息需处理
- 0分：存在严重敏感信息泄露，需退回修改

### 4.2 授权声明（5分）

| 检查项 | 分值 | 要求 |
|--------|------|------|
| 公开级别声明 | 3分 | 明确选择公开/半匿名/匿名 |
| 授权范围说明 | 2分 | 说明授权项目和范围 |

**评分标准**：

- 5分：授权声明完整明确
- 3-4分：授权声明基本完整
- 1-2分：授权声明不完整
- 0分：无授权声明

### 4.3 合规性（5分）

| 检查项 | 分值 | 要求 |
|--------|------|------|
| 符合公司政策 | 3分 | 内容符合提交者公司分享政策 |
| 无商业机密 | 2分 | 不包含商业敏感信息 |

**评分标准**：

- 5分：完全合规
- 3-4分：基本合规，需轻微调整
- 1-2分：存在合规风险
- 0分：严重不合规，需退回

## 审核结论类型

### ✅ 通过（总分 ≥ 70，单项 ≥ 50）

案例质量优秀，可直接收录发布。

### 📝 修改后通过（总分 60-69，或单项 40-49）

案例基本合格，需根据反馈进行修改后再次提交。

**常见修改建议**：

- 补充缺失的技术细节
- 完善性能数据
- 进一步脱敏处理
- 优化架构图说明

### ❌ 不通过（总分 < 60，或单项 < 40）

案例存在严重问题，建议重新整理后提交。

**常见不通过原因**：

- 内容过于简单，缺乏技术深度
- 数据明显不真实
- 存在严重敏感信息泄露
- 与流计算主题无关

## 审核流程说明

### 格式审核（1-2个工作日）

- 检查模板字段完整性
- 验证必填项是否已填写
- 初步检查敏感信息
- **审核人**：项目维护团队成员

### 技术审核（3-5个工作日）

- 技术专家验证架构合理性
- 评估技术方案的创新性和参考价值
- 确认性能数据的可信度
- **审核人**：流计算领域技术专家

### 隐私审核（1-2个工作日）

- 全面检查敏感信息脱敏情况
- 确认授权声明完整
- 验证公开级别设置
- **审核人**：项目维护团队负责人

---

## 特殊情况处理

### 匿名案例

- 隐私合规评分自动满分（20分）
- 重点审核技术价值和数据合理性
- 可接受相对较少的数据细节

### 知名公司案例

- 可适当降低数据完整性要求
- 重点关注技术方案的创新性
- 隐私审核需更加严格

### 个人贡献者案例

- 可适当降低部分要求
- 重点关注技术深度和经验总结
- 提供案例撰写指导
