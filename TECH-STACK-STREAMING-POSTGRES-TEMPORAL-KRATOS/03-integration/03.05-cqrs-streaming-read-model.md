# CQRS + 流式读模型

> 所属阶段: TECH-STACK | 前置依赖: [02.01-postgresql-18-cdc-deep-dive.md, 02.04-flink-streaming-resilience.md] | 形式化等级: L4

## 1. 概念定义 (Definitions)

**Def-T-03-05-01 (CQRS — Command Query Responsibility Segregation)**

CQRS 是一种架构模式，将系统的**命令（写操作）**与**查询（读操作）**在模型层面彻底分离。写模型负责状态变更的验证与持久化，读模型负责面向查询优化的数据投影。形式上，设系统状态空间为 $S$，命令集合为 $C$，查询集合为 $Q$：

$$
\text{CQRS}(S, C, Q) \triangleq \langle W : C \to S \to S,\ R : S \to Q \to V,\ \pi : S_W \xrightarrow{\text{async}} S_R \rangle
$$

其中 $W$ 为写处理函数，$R$ 为读处理函数，$\pi$ 为从写模型状态 $S_W$ 到读模型状态 $S_R$ 的异步投影映射。

**Def-T-03-05-02 (写模型 — Write Model)**

写模型是 CQRS 中负责处理状态变更命令的持久化层。在本技术栈中，写模型为 **PostgreSQL 18** 标准化表结构，支持完整 ACID 事务语义。写模型 Schema 严格遵循数据库范式（通常 3NF/BCNF），以消除更新异常并保证写入一致性。形式化地：

$$
\text{WriteModel} \triangleq \langle T, \mathcal{ACID}, \Phi_{\text{constraint}} \rangle
$$

其中 $T$ 为事务操作序列，$\mathcal{ACID}$ 为原子性、一致性、隔离性、持久性保证，$\Phi_{\text{constraint}}$ 为外键、CHECK、唯一约束集合。

**Def-T-03-05-03 (读模型 — Read Model)**

读模型是 CQRS 中面向查询优化的数据投影，通常以反规范化（denormalized）、预聚合（pre-aggregated）的形式存在。读模型与写模型 Schema 解耦，允许为特定查询场景独立设计存储结构。形式化地：

$$
\text{ReadModel} \triangleq \langle S_R, Q_{\text{opt}}, \mathcal{V}_{\text{stale}} \rangle
$$

其中 $S_R$ 为读模型状态，$Q_{\text{opt}}$ 为针对目标查询优化的索引与存储布局，$\mathcal{V}_{\text{stale}}$ 为允许读取过时版本的谓词。

**Def-T-03-05-04 (物化视图 — Materialized View)**

物化视图是将查询结果预先计算并持久化的数据结构。在流式 CQRS 架构中，物化视图由 Flink 作业从 CDC 事件流实时构建并持续更新，而非通过数据库原生 `MATERIALIZED VIEW` 机制刷新。形式化地：

$$
\text{MaterializedView}(t) \triangleq \bigoplus_{e \in \text{CDC}[0,t]} f_{\text{agg}}(e)
$$

其中 $\text{CDC}[0,t]$ 为从起始时刻到 $t$ 的变更数据捕获事件序列，$f_{\text{agg}}$ 为聚合函数（如 `SUM`、`COUNT`、`AVG` 或自定义窗口聚合），$\bigoplus$ 为聚合状态的累积算子。

**Def-T-03-05-05 (最终一致性延迟 — Eventual Consistency Lag)**

最终一致性延迟 $\delta$ 定义为写模型事务提交时刻 $t_{\text{commit}}$ 到读模型反映该变更的最早可查询时刻 $t_{\text{visible}}$ 之间的时间间隔：

$$
\delta \triangleq t_{\text{visible}} - t_{\text{commit}}
$$

在流式 CQRS 架构中，$\delta$ 由以下子延迟组成：

$$
\delta = \delta_{\text{wal}} + \delta_{\text{debezium}} + \delta_{\text{kafka}} + \delta_{\text{flink}} + \delta_{\text{sink}}
$$

分别对应：WAL 刷盘延迟、Debezium 捕获延迟、Kafka 传输延迟、Flink 处理延迟、Sink 写入延迟。

## 2. 属性推导 (Properties)

**Prop-T-03-05-01 (读写分离的故障隔离性)**

在 CQRS 架构中，写模型与读模型运行于独立的故障域（fault domain）。设写模型可用性为 $A_W$，读模型可用性为 $A_R$，则系统整体查询可用性 $A_{\text{query}}$ 满足：

$$
A_{\text{query}} = A_R \perp A_W
$$

即读模型的可用性不依赖于写模型的可用性（符号 $\perp$ 表示故障独立性）。当写模型发生故障时，读模型仍可基于已物化的历史视图提供只读查询服务；反之，读模型的重新计算或延迟激增不会影响写模型的写入吞吐。

> 直观解释：写模型 PG18 宕机时，Flink 已构建的读模型（如 Redis 中的聚合缓存）仍然可查询；Flink 作业重启时，PG18 的订单写入不受影响。

**Lemma-T-03-05-01 (读模型延迟上界)**

假设 Flink 作业以事件时间处理 CDC 流，且使用允许的延迟（allowed lateness）策略 $L$，则读模型反映写模型变更的延迟上界为：

$$
\delta_{\max} = L + T_{\text{window}} + \delta_{\text{network}}
$$

其中 $T_{\text{window}}$ 为窗口大小（TUMBLE/HOP/SESSION），$\delta_{\text{network}}$ 为网络传输的最大延迟。对于无窗口的逐事件物化视图（per-event materialization），上界简化为：

$$
\delta_{\max}^{\text{event}} = \delta_{\text{network}} + \delta_{\text{state-backend}}
$$

其中 $\delta_{\text{state-backend}}$ 为 Flink 状态后端（RocksDB/Heap）的写入延迟。

**Proof sketch**: Flink 水印机制保证事件时间 $t$ 的所有事件在水印 $W(t)$ 推进后被处理。设水印最大滞后为 $L$，则窗口 $[t, t+T_{\text{window}})$ 的结果最晚在 $t + T_{\text{window}} + L$ 时刻输出。叠加网络传输延迟即得上界。$\square$

## 3. 关系建立 (Relations)

### 3.1 CQRS 与事件溯源 (Event Sourcing)

事件溯源将系统状态建模为不可变事件序列的左折叠（left fold），CQRS 将读写分离。两者天然互补：

- **事件溯源作为写模型**：写模型不存储当前状态，而是存储导致状态变更的领域事件序列。PG18 的 CDC 变更记录可视为轻量级事件溯源的实现。
- **CQRS 作为投影机制**：读模型是事件流通过投影函数 $P$ 的物化结果：$S_R = P(\text{events})$。
- **差异**：经典事件溯源要求命令处理时从事件流重放状态；本技术栈中 PG18 仍维护当前状态表，CDC 仅作为变更传播机制，降低了重放开销。

### 3.2 CQRS 与 CDC (Change Data Capture)

CDC 是连接写模型与读模型的**异步数据管道**。Debezium 从 PG18 WAL 捕获变更事件，将写模型的内部事务日志转化为领域无关的事件流：

$$
\text{CDC} : \Delta S_W \xrightarrow{\text{Debezium}} \text{Kafka Topic} \xrightarrow{\text{Flink}} \Delta S_R
$$

关系特性：

- **解耦**：写模型无需感知读模型的存在，读模型的增删改不影响写模型 Schema。
- **有序性**：PG18 事务提交顺序通过 Kafka 分区保序机制传递到 Flink。
- **完整性**：Debezium 快照 + 流模式保证不遗漏历史数据。

### 3.3 CQRS 与 Flink 流处理

Flink 充当读模型的**实时计算引擎**和**状态管理器**：

| 维度 | 写模型 (PG18) | Flink 读模型构建 |
|------|--------------|-----------------|
| 一致性模型 | 强一致性 (ACID) | 最终一致性 |
| 延迟 | 毫秒级事务提交 | 秒级 ~ 分钟级（取决于窗口） |
| 查询能力 | 规范化表 JOIN | 预聚合反规范化视图 |
| 扩展性 | 垂直扩展为主 | 水平扩展（并行度可调） |
| 容错 | WAL + 物理复制 | Checkpoint + Savepoint |

Flink 的窗口聚合能力（`TUMBLE`、`HOP`、`SESSION`、`CUMULATE`）使读模型能够表达复杂的时间维度分析查询，这些查询若在 PG18 上实时执行将产生极高的 OLTP 负载。

## 4. 论证过程 (Argumentation)

### 4.1 写模型：PG18 作为事务性写数据库

PostgreSQL 18 承担 CQRS 写模型的核心职责，其设计原则如下：

- **Schema 规范化**：订单表、用户表、库存表等严格遵循 3NF，通过外键约束维护引用完整性，避免更新异常。
- **ACID 事务**：订单创建涉及库存扣减、余额扣减、订单记录插入等多表操作，必须在一个事务内原子完成。
- **低延迟写入**：通过 `synchronous_commit = local` 或 `off`（视 durability 要求而定）平衡持久性与写入延迟。
- **虚拟生成列 (Generated Columns)**：PG18 支持 `GENERATED ALWAYS AS` 虚拟列，可用于在不修改写模型核心 Schema 的前提下，派生读模型需要的计算字段（如 `total_amount_after_tax`），避免应用层重复计算。

```sql
-- 写模型示例：订单表（规范化）
CREATE TABLE orders (
    order_id        BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL REFERENCES users(user_id),
    region_code     VARCHAR(8) NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    raw_amount      NUMERIC(18,4) NOT NULL,
    -- 虚拟生成列：读模型派生字段，不占用存储（STORED 则持久化）
    tax_amount      NUMERIC(18,4) GENERATED ALWAYS AS (raw_amount * 0.08) STORED,
    total_amount    NUMERIC(18,4) GENERATED ALWAYS AS (raw_amount * 1.08) STORED
);
```

### 4.2 读模型：Flink 从 Kafka CDC 流构建物化视图

读模型通过 Flink SQL/DataStream 从 Debezium → Kafka 的 CDC 流实时构建：

```sql
-- Flink SQL：按地区/小时窗口的订单统计物化视图
CREATE TABLE order_cdc (
    order_id        BIGINT,
    region_code     STRING,
    total_amount    DECIMAL(18,4),
    event_time      TIMESTAMP_LTZ(3),
    -- Debezium 元数据
    op              STRING,
    ts_ms           BIGINT,
    WATERMARK FOR event_time AS event_time - INTERVAL '5' SECOND
) WITH (
    'connector' = 'kafka',
    'topic' = 'db.public.orders',
    'format' = 'debezium-json',
    ...
);

CREATE TABLE region_order_stats (
    region_code     STRING,
    window_start    TIMESTAMP_LTZ(3),
    window_end      TIMESTAMP_LTZ(3),
    order_count     BIGINT,
    total_amount    DECIMAL(18,4),
    PRIMARY KEY (region_code, window_start) NOT ENFORCED
) WITH (
    'connector' = 'jdbc',
    'url' = 'jdbc:postgresql://read-db:5432/analytics',
    'table-name' = 'region_order_stats'
);

-- TUMBLE 窗口聚合 → 物化视图
INSERT INTO region_order_stats
SELECT
    region_code,
    TUMBLE_START(event_time, INTERVAL '1' HOUR) AS window_start,
    TUMBLE_END(event_time, INTERVAL '1' HOUR) AS window_end,
    COUNT(*) AS order_count,
    SUM(total_amount) AS total_amount
FROM order_cdc
WHERE op <> 'd'  -- 排除 DELETE 操作或根据业务处理
GROUP BY
    region_code,
    TUMBLE(event_time, INTERVAL '1' HOUR);
```

读模型存储可选 PostgreSQL（复杂查询）、Elasticsearch（全文检索）、Redis（低延迟 KV）或 Apache Pinot（OLAP），根据查询 SLA 独立选型。

### 4.3 读写分离的故障隔离

CQRS 的故障隔离体现在两个维度：

**维度一：写故障不影响读查询**

- PG18 主库宕机时，读模型存储（如 Redis/ES）中的物化视图仍在有效期内。
- 读模型可配置 TTL（Time-To-Live），在写模型恢复前提供带 `stale` 标记的降级数据。

**维度二：读延迟不影响写吞吐**

- Flink 作业重启、反压或窗口聚合计算过载时，不会阻塞 PG18 的写入事务。
- Kafka 作为缓冲层解耦了生产速率（PG18 CDC）与消费速率（Flink 处理）。

形式化描述：设故障集合为 $F = \{f_W, f_R\}$，其中 $f_W$ 为写模型故障，$f_R$ 为读模型故障。CQRS 的故障隔离保证：

$$
\forall f_W \in F_W,\ R(S_R) \text{ 仍可查询（可能返回 stale 数据）}
$$
$$
\forall f_R \in F_R,\ W(C, S_W) \text{ 仍可提交事务}
$$

### 4.4 组合弹性 (Composite Resilience)

#### 4.4.1 读模型延迟的降级策略

当 $\delta > \delta_{\text{SLA}}$（SLA 阈值，如 30 秒）时，系统进入降级模式：

1. **返回 Stale 数据**：读模型存储保留最后一次成功 Sink 的快照，查询接口通过 HTTP Header `X-Data-Staleness: true` 或响应字段 `stale: true` 标记。
2. **延迟指标暴露**：Flink 作业通过 Prometheus 暴露 `kafka-lag`、`checkpoint-duration`、`numRecordsInPerSecond` 等指标，供监控系统判断读模型健康度。
3. **自动熔断**：当 Kafka Lag 超过阈值 $L_{\max}$ 时，API Gateway 自动切换为只读降级模式，避免向客户端返回部分聚合的不一致结果。

#### 4.4.2 Flink 状态丢失后的重建

Flink 作业状态（如窗口聚合中间状态）存储于 RocksDB/Heap，通过 Checkpoint 定期持久化到 HDFS/S3。若状态丢失（如状态后端损坏）：

1. **从 Kafka 最早 Offset 重放**：Flink 作业配置 `scan.startup.mode = earliest-offset`，从 Kafka Topic 起始位置重新消费 CDC 事件。
2. **幂等 Sink**：读模型存储的主键设计为 `(region_code, window_start)`，确保重放时同一窗口结果覆盖写入，不产生重复。
3. **增量快照（Incremental Checkpoint）**：RocksDB 增量 Checkpoint 减少状态恢复时间，结合 Local Recovery 加速 TaskManager 重启。

恢复时间目标（RTO）上界：

$$
\text{RTO}_{\max} = T_{\text{restart}} + \frac{V_{\text{state}}}{B_{\text{restore}}} + \delta_{\text{catch-up}}
$$

其中 $V_{\text{state}}$ 为状态体积，$B_{\text{restore}}$ 为状态恢复带宽，$\delta_{\text{catch-up}}$ 为 Kafka Lag 追赶时间。

#### 4.4.3 读模型多副本

读模型存储采用多副本部署提升查询可用性：

- **PostgreSQL 读副本**：通过物理复制或逻辑复制创建只读副本，分担分析查询负载。
- **Redis Cluster**：分片存储热点聚合结果，支持主从故障自动切换。
- **Elasticsearch 副本分片**：默认配置 1 主 + 1~2 副本，查询请求分散到多个节点。

多副本下读模型可用性：

$$
A_R = 1 - \prod_{i=1}^{n} (1 - A_{R_i})
$$

其中 $A_{R_i}$ 为第 $i$ 个副本的可用性，$n$ 为副本数。

## 5. 形式证明 / 工程论证 (Proof / Engineering Argument)

### 5.1 定理：CQRS 在最终一致性模型下的可用性提升

**Thm-T-03-05-01**

设单体架构（非 CQRS）的读写可用性为 $A_{\text{mono}}$，CQRS 架构的读可用性为 $A_R$、写可用性为 $A_W$。在最终一致性模型下，若读模型与写模型故障独立，且读模型允许返回 stale 数据，则 CQRS 的查询可用性严格优于单体架构：

$$
A_{\text{CQRS-query}} > A_{\text{mono}}
$$

**Proof.**

设单体架构中读写共享同一数据库实例，其可用性受限于该实例的故障率 $\lambda$ 和恢复率 $\mu$：

$$
A_{\text{mono}} = \frac{\mu}{\lambda + \mu}
$$

在 CQRS 架构中，写模型 PG18 的可用性为 $A_W = \frac{\mu_W}{\lambda_W + \mu_W}$，读模型（如 Redis Cluster）的可用性为 $A_R = \frac{\mu_R}{\lambda_R + \mu_R}$。由于故障独立，读模型查询可用性为：

$$
A_{\text{CQRS-query}} = A_R + (1 - A_R) \cdot A_W \cdot p_{\text{fallback}}
$$

其中 $p_{\text{fallback}}$ 为写模型可用时通过同步查询提供读服务的能力。更关键的是，当引入 stale 读取降级策略后，即使 $A_R$ 暂时下降（Flink 作业重启），读模型存储中的历史快照仍可查询，等效于读模型具有更高的故障容忍度。

具体地，设 Flink 作业故障导致读模型更新中断的概率为 $p_{\text{lag}}$，但读模型存储（Redis/ES）自身可用性 $A_{\text{store}}$ 独立于 Flink。此时有效读可用性为：

$$
A_{\text{effective}} = A_{\text{store}} \cdot (1 - p_{\text{lag}} \cdot (1 - \alpha))
$$

其中 $\alpha \in [0,1]$ 为 stale 数据可接受度（$\alpha = 1$ 表示完全接受 stale 数据，读可用性等于存储可用性）。由于单体架构无此解耦，$A_{\text{mono}} = A_W = A_{\text{store}}^{\text{mono}}$，且单体存储同时承担读写压力，其故障率 $\lambda^{\text{mono}}$ 通常高于专门的读模型存储故障率 $\lambda_R$（因读模型可选择更易扩展的存储如 Redis Cluster）。

因此：

$$
A_{\text{effective}} \geq A_{\text{store}} > A_{\text{mono}} \quad \text{当 } \lambda_R < \lambda^{\text{mono}} \text{ 且 } \alpha > 0
$$

$\square$

### 5.2 工程论证：最终一致性下的业务可接受性

最终一致性并非"弱一致性"的同义词，而是在特定业务边界内的**有界延迟一致性**。论证 CQRS 适用于以下业务条件：

1. **读操作远多于写操作**：电商场景中订单查询/统计查询频率比下单频率高 2~3 个数量级，读写分离将读负载从 PG18 卸载到 Flink 物化视图，避免 OLTP 数据库成为瓶颈。

2. **业务容忍分钟级延迟**：订单统计报表、地区销售额看板等分析型查询通常不需要秒级实时性，TUMBLE/HOP 窗口的延迟上界（Lemma-T-03-05-01）在业务可接受范围内。

3. **幂等补偿机制**：若读模型 transiently 不一致（如 Flink 作业重启期间窗口聚合缺失），业务层可通过异步对账任务检测并补偿，不影响核心交易链路。

## 6. 实例验证 (Examples)

### 6.1 电商场景：订单写表 + 实时订单统计物化视图

**场景描述**：某跨境电商平台需要实时统计各地区的每小时订单量与销售额，用于运营看板与库存预调度。订单写入通过 PG18 事务完成，统计视图通过 Flink 实时构建。

**写模型（PG18）**：

```sql
-- 写模型：标准化订单表
CREATE TABLE orders (
    order_id        BIGSERIAL PRIMARY KEY,
    user_id         BIGINT NOT NULL,
    region_code     VARCHAR(8) NOT NULL,  -- 'NA-EAST', 'EU-WEST', 'APAC-SG'
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    currency        VARCHAR(3) NOT NULL,
    raw_amount      NUMERIC(18,4) NOT NULL,
    tax_amount      NUMERIC(18,4) GENERATED ALWAYS AS (raw_amount * tax_rate) STORED,
    total_amount    NUMERIC(18,4) GENERATED ALWAYS AS (raw_amount + tax_amount) STORED,
    status          VARCHAR(16) NOT NULL DEFAULT 'PENDING',

    CONSTRAINT chk_region CHECK (region_code ~ '^[A-Z]{2}-[A-Z]+$'),
    CONSTRAINT chk_amount_positive CHECK (raw_amount > 0)
);

CREATE INDEX idx_orders_created_at ON orders(created_at);
CREATE INDEX idx_orders_region ON orders(region_code, created_at);
```

**读模型构建（Flink SQL）**：

```sql
-- CDC 源表
CREATE TABLE orders_cdc (
    order_id        BIGINT,
    region_code     STRING,
    total_amount    DECIMAL(18,4),
    created_at      TIMESTAMP_LTZ(3),
    op              STRING,  -- c=create, u=update, d=delete
    WATERMARK FOR created_at AS created_at - INTERVAL '10' SECOND
) WITH (
    'connector' = 'kafka',
    'topic' = 'db.public.orders',
    'properties.bootstrap.servers' = 'kafka:9092',
    'format' = 'debezium-json',
    'scan.startup.mode' = 'latest-offset'
);

-- 读模型 Sink：PostgreSQL 分析库
CREATE TABLE hourly_region_stats (
    region_code     STRING,
    window_hour     TIMESTAMP_LTZ(3),
    order_count     BIGINT,
    gmv             DECIMAL(18,4),
    avg_order_value DECIMAL(18,4),
    PRIMARY KEY (region_code, window_hour) NOT ENFORCED
) WITH (
    'connector' = 'jdbc',
    'url' = 'jdbc:postgresql://pg18-analytics:5432/reporting',
    'table-name' = 'hourly_region_stats',
    'username' = 'flink_sink',
    'password' = '${FLINK_JDBC_PASSWORD}'
);

-- 物化视图：每小时地区统计
INSERT INTO hourly_region_stats
SELECT
    region_code,
    TUMBLE_START(created_at, INTERVAL '1' HOUR) AS window_hour,
    COUNT(*) AS order_count,
    SUM(total_amount) AS gmv,
    AVG(total_amount) AS avg_order_value
FROM orders_cdc
WHERE op IN ('c', 'u')  -- 仅处理创建和更新
GROUP BY
    region_code,
    TUMBLE(created_at, INTERVAL '1' HOUR);
```

**降级与监控**：

```java
// 读模型查询服务（Spring Boot）
@GetMapping("/api/v1/stats/hourly")
public ResponseEntity<HourlyStats> getHourlyStats(
        @RequestParam String regionCode,
        @RequestParam @DateTimeFormat(iso = DateTimeFormat.ISO.DATE_TIME) Instant hour) {

    HourlyStats stats = readModelRepository.findByRegionAndHour(regionCode, hour);

    // 检查 Flink 作业健康度（通过 Prometheus 查询）
    boolean isFlinkHealthy = metricsClient.isLagWithinThreshold("flink.orders.cdc", Duration.ofSeconds(30));

    if (!isFlinkHealthy) {
        // 返回 stale 数据并标记
        return ResponseEntity.ok()
            .header("X-Data-Staleness", "true")
            .header("X-Staleness-Reason", "Flink job lag exceeds threshold")
            .body(stats);
    }

    return ResponseEntity.ok(stats);
}
```
