实验4: 状态TTL与过期策略可视化

通过交互式演示理解状态生命周期管理和TTL清理机制

📖 实验原理

什么是状态TTL (Time-To-Live)?

状态TTL允许为状态设置过期时间，当状态超过TTL未被更新时，会被标记为过期。Flink提供多种清理策略来管理过期状态，防止状态无限增长。

🕐 TTL 配置参数

  • ttlDuration: 状态过期时间
  • updateType: 更新类型 (OnCreateAndWrite / OnReadAndWrite)
  • stateVisibility: 过期状态可见性 (ReturnExpiredIfNotCleanedUp / NeverReturnExpired)

🧹 清理策略

  • CleanupInBackground: 后台异步清理 (默认)
  • CleanupIncrementally: 读取时增量清理
  • CleanupInRocksdbCompactFilter: RocksDB压缩时清理

🎯 观察要点

实验中请关注:

  • 状态如何随时间从绿色(活跃) → 橙色(将过期) → 红色(已过期) → 灰色(已清理)
  • 不同更新策略下，访问操作如何延长状态生命周期
  • 清理策略对内存占用的影响
  • TTL时长设置与状态数量的关系

✏️ 实验任务

任务列表

  1. 设置TTL为10秒，观察状态从创建到清理的完整生命周期
  2. 切换更新策略为 OnReadAndWrite，观察访问如何延长TTL
  3. 比较不同清理策略下内存使用的差异
  4. 思考：在实际应用中，如何选择合适的TTL和清理策略？

💻 代码参考

// TTL 配置示例
StateTtlConfig ttlConfig = StateTtlConfig
    .newBuilder(Time.hours(24))
    .setUpdateType(
        StateTtlConfig.UpdateType.OnCreateAndWrite
    )
    .setStateVisibility(
        StateTtlConfig.StateVisibility.NeverReturnExpired
    )
    .cleanupIncrementally(10, true)
    .build();

ValueStateDescriptor<MyState> descriptor = 
    new ValueStateDescriptor<>("myState", MyState.class);
descriptor.enableTimeToLive(ttlConfig);
