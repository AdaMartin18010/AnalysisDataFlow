5.1.1 状态类型概述

Flink 状态类型分类:
┌─────────────────────────────────────────────────────────────┐
│  Keyed State (键控状态)                                      │
│  ├─ ValueState      - 单值状态                            │
│  ├─ ListState       - 列表状态                            │
│  ├─ MapState   - Map 结构状态                        │
│  ├─ ReducingState   - 用于聚合的归约状态                   │
│  └─ AggregatingState - 用于聚合的聚合状态            │
│                                                              │
│  Operator State (算子状态)                                   │
│  ├─ ListState       - 列表状态                            │
│  └─ BroadcastState - 广播状态                          │
│                                                              │
│  State Backend (状态后端)                                    │
│  ├─ MemoryStateBackend    - 内存存储 (测试用)                 │
│  ├─ FsStateBackend        - 文件系统存储                     │
│  └─ RocksDBStateBackend   - RocksDB 存储 (推荐生产)           │
└─────────────────────────────────────────────────────────────┘

5.1.2 Keyed State 使用

public class CountWithKeyedState extends KeyedProcessFunction<String, Event, Result> {
    
    // 声明 ValueState
    private ValueState<Long> countState;
    private ValueState<Long> lastTimestampState;
    
    @Override
    public void open(Configuration parameters) {
        // 初始化状态描述符
        StateTtlConfig ttlConfig = StateTtlConfig
            .newBuilder(Time.hours(24))
            .setUpdateType(StateTtlConfig.UpdateType.OnCreateAndWrite)
            .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired)
            .build();
        
        ValueStateDescriptor<Long> countDescriptor = 
            new ValueStateDescriptor<>("count", Types.LONG);
        countDescriptor.enableTimeToLive(ttlConfig);
        countState = getRuntimeContext().getState(countDescriptor);
        
        ValueStateDescriptor<Long> timestampDescriptor = 
            new ValueStateDescriptor<>("lastTimestamp", Types.LONG);
        lastTimestampState = getRuntimeContext().getState(timestampDescriptor);
    }
    
    @Override
    public void processElement(Event event, Context ctx, Collector<Result> out) 
            throws Exception {
        // 获取当前状态值
        Long currentCount = countState.value();
        if (currentCount == null) {
            currentCount = 0L;
        }
        
        // 更新状态
        currentCount++;
        countState.update(currentCount);
        lastTimestampState.update(event.getTimestamp());
        
        // 输出结果
        out.collect(new Result(ctx.getCurrentKey(), currentCount));
    }
}

// 使用
stream.keyBy(Event::getUserId)
    .process(new CountWithKeyedState())
    .print();

5.1.3 MapState 使用示例

public class UserSessionTracker extends KeyedProcessFunction<String, Event, SessionInfo> {
    
    private MapState<String, Long> pageVisitState;
    private ValueState<Long> sessionStartState;
    
    @Override
    public void open(Configuration parameters) {
        MapStateDescriptor<String, Long> pageDescriptor = 
            new MapStateDescriptor<>(
                "pageVisits", 
                Types.STRING, 
                Types.LONG
            );
        pageVisitState = getRuntimeContext().getMapState(pageDescriptor);
        
        ValueStateDescriptor<Long> sessionDescriptor = 
            new ValueStateDescriptor<>("sessionStart", Types.LONG);
        sessionStartState = getRuntimeContext().getState(sessionDescriptor);
    }
    
    @Override
    public void processElement(Event event, Context ctx, Collector<SessionInfo> out) 
            throws Exception {
        String page = event.getPage();
        
        // 更新页面访问计数
        Long count = pageVisitState.get(page);
        if (count == null) {
            count = 0L;
        }
        pageVisitState.put(page, count + 1);
        
        // 检查会话开始时间
        if (sessionStartState.value() == null) {
            sessionStartState.update(event.getTimestamp());
        }
        
        // 注册定时器用于会话超时检测
        ctx.timerService().registerEventTimeTimer(
            event.getTimestamp() + Time.minutes(30).toMilliseconds()
        );
    }
    
    @Override
    public void onTimer(long timestamp, OnTimerContext ctx, Collector<SessionInfo> out) 
            throws Exception {
        // 会话超时，输出会话统计
        Long sessionStart = sessionStartState.value();
        if (sessionStart != null) {
            Map<String, Long> pageStats = new HashMap<>();
            for (Map.Entry<String, Long> entry : pageVisitState.entries()) {
                pageStats.put(entry.getKey(), entry.getValue());
            }
            
            out.collect(new SessionInfo(
                ctx.getCurrentKey(),
                sessionStart,
                timestamp,
                pageStats
            ));
            
            // 清理状态
            pageVisitState.clear();
            sessionStartState.clear();
        }
    }
}

5.2 Checkpoint 机制详解

🎯 学习目标

  • 深入理解 Checkpoint 的工作原理

  • 理解对齐 Checkpoint 与非对齐 Checkpoint

5.2.1 Checkpoint 流程

Checkpoint 执行流程:
┌─────────────────────────────────────────────────────────────────┐
│ 1. JM 触发 Checkpoint (定期或手动)                                │
│        ↓                                                        │
│ 2. JM 向 Source 发送 Checkpoint Barrier                          │
│        ↓                                                        │
│ 3. Source 注入 Barrier，保存偏移量，向下游广播                      │
│        ↓                                                        │
│ 4. 算子接收到 Barrier 后:                                        │
│    ├─ 对齐模式: 阻塞其他输入，等待所有 Barrier 到达                 │
│    └─ 快照状态，向下游广播 Barrier                                │
│        ↓                                                        │
│ 5. Sink 确认 Checkpoint 完成                                     │
│        ↓                                                        │
│ 6. JM 收集所有确认，标记 Checkpoint 完成                           │
└─────────────────────────────────────────────────────────────────┘

Barrier 对齐示意 (双输入算子):
输入1: [A1]──[A2]──[Barrier]──[A3]──[A4]
                              ↓ (等待)
输入2: [B1]──[Barrier]────────[B2]──[B3]
       
对齐后: 先处理到 Barrier 的数据，对齐后快照状态

5.2.2 Checkpoint 配置

// Checkpoint 基础配置
StreamExecutionEnvironment env = 
    StreamExecutionEnvironment.getExecutionEnvironment();

// 启用 Checkpoint，每60秒触发一次
env.enableCheckpointing(60000);

// 高级配置
CheckpointConfig checkpointConfig = env.getCheckpointConfig();

// 模式: EXACTLY_ONCE 或 AT_LEAST_ONCE
checkpointConfig.setCheckpointingMode(
    CheckpointingMode.EXACTLY_ONCE
);

// 超时时间 (Checkpoint 超过此时长会被丢弃)
checkpointConfig.setCheckpointTimeout(600000); // 10分钟

// 最小间隔 (两次 Checkpoint 之间的最小间隔)
checkpointConfig.setMinPauseBetweenCheckpoints(30000); // 30秒

// 最大并发 Checkpoint 数
checkpointConfig.setMaxConcurrentCheckpoints(1);

// 外部化 Checkpoint (作业取消后仍保留)
checkpointConfig.enableExternalizedCheckpoints(
    ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION
);

// 非对齐 Checkpoint (Flink 1.11+)
checkpointConfig.enableUnalignedCheckpoints();
checkpointConfig.setAlignmentTimeout(Duration.ofSeconds(30));

5.2.4 非对齐 Checkpoint

// 非对齐 Checkpoint 配置
// 适用场景: 高背压、低延迟要求、大并行度

checkpointConfig.enableUnalignedCheckpoints();

// 对齐超时 (超过此时间自动切换为非对齐)
checkpointConfig.setAlignmentTimeout(Duration.ofSeconds(10));

// 非对齐 Checkpoint 原理:
// 1. Barrier 不等待对齐，立即向下游传递
// 2. 将缓冲的数据作为状态的一部分保存
// 3. 恢复时从状态恢复缓冲数据

// 内存限制 (控制非对齐 Checkpoint 的最大内存使用)
checkpointConfig.setAlignedCheckpointTimeout(Duration.ofMinutes(1));

// 注意: 非对齐 Checkpoint 会增加状态大小
// 适合: 背压严重、端到端延迟敏感的场景
// 不适合: 网络带宽受限、小缓冲区的场景

5.3 Exactly-Once 语义实现

🎯 学习目标

  • 理解一致性级别的概念 (At-Most-Once, At-Least-Once, Exactly-Once)
  • 掌握端到端 Exactly-Once 的实现原理
  • 学会使用 Two-Phase Commit
  • 理解幂等写入和事务性写入

5.3.1 一致性级别对比

一致性级别对比:
┌─────────────────────────────────────────────────────────────────┐
│ At-Most-Once (最多一次)                                          │
│ ├─ 特点: 不保证数据到达，可能丢失                                 │
│ ├─ 实现: 不开启 Checkpoint，故障时从最新位置开始                  │
│ └─ 适用: 允许数据丢失的统计类场景                                  │
│                                                                 │
│ At-Least-Once (至少一次)                                         │
│ ├─ 特点: 保证数据不丢失，可能重复                                 │
│ ├─ 实现: 开启 Checkpoint，故障时从上次 Checkpoint 恢复            │
│ └─ 适用: 幂等写入或允许重复的场景                                  │
│                                                                 │
│ Exactly-Once (精确一次)                                          │
│ ├─ 特点: 保证数据不丢失且不重复                                   │
│ ├─ 实现: 两阶段提交 + 事务性 Sink                                 │
│ └─ 适用: 金融交易、订单处理等对准确性要求高的场景                    │
└─────────────────────────────────────────────────────────────────┘

端到端 Exactly-Once 条件:
1. Source 可重放 (如 Kafka 可重置 offset)
2. Flink 内部 Exactly-Once (Checkpoint)
3. Sink 支持事务或幂等写入

5.3.2 Two-Phase Commit

Two-Phase Commit (2PC) 协议:
┌─────────────────────────────────────────────────────────────────┐
│ 预提交阶段 (Pre-commit)                                          │
│ 1. Flink 开始 Checkpoint                                          │
│ 2. Sink 收到 Barrier，将数据写入外部系统但不提交                   │
│ 3. Sink 将预提交状态保存到 Checkpoint                             │
│        ↓                                                        │
│ 提交阶段 (Commit)                                                │
│ 4. JM 确认所有算子 Checkpoint 完成                                │
│ 5. 通知 Sink 提交事务                                             │
│ 6. 外部系统数据正式可见                                           │
│        ↓                                                        │
│ 故障恢复                                                         │
│ 7. 如果 Checkpoint 失败，回滚预提交的事务                          │
│ 8. 从上一个成功 Checkpoint 恢复                                   │
└─────────────────────────────────────────────────────────────────┘

Kafka Producer 事务示例:
事务ID: flink-job-{jobId}-{subtaskId}

props.put("bootstrap.servers", "localhost:9092");
props.put("transaction.timeout.ms", "900000"); // 15分钟

    "output-topic",                              // 目标 topic

    FlinkKafkaProducer.Semantic.EXACTLY_ONCE     // 语义级别
);

// 自定义 TwoPhaseCommitSinkFunction
public class ExactlyOnceFileSink extends TwoPhaseCommitSinkFunction<Event, 
        ExactlyOnceFileSink.Transaction, ExactlyOnceFileSink.Transaction> {
    
    private transient FileSystem fs;
    
    public ExactlyOnceFileSink() {
        super(
            TypeInformation.of(Event.class).createSerializer(new ExecutionConfig()),
            TypeInformation.of(Transaction.class).createSerializer(new ExecutionConfig())
        );
    }
    
    @Override
    protected void invoke(Transaction transaction, Event event, Context context) {
        // 写入临时文件
        transaction.write(event);
    }
    
    @Override
    protected Transaction beginTransaction() {
        // 创建临时事务文件
        String tempFile = "/tmp/" + UUID.randomUUID();
        return new Transaction(tempFile);
    }
    
    @Override
    protected void preCommit(Transaction transaction) {
        // 刷盘，确保数据持久化
        transaction.flush();
    }
    
    @Override
    protected void commit(Transaction transaction) {
        // 将临时文件移动到正式目录
        transaction.commitToFinalLocation();
    }
    
    @Override
    protected void abort(Transaction transaction) {
        // 删除临时文件
        transaction.rollback();
    }
}

// 幂等写入示例: Elasticsearch 按文档ID更新
ElasticsearchSink.Builder<Event> builder = new ElasticsearchSink.Builder<>(
    hosts,
    new ElasticsearchSinkFunction<Event>() {

        }
    }

// Redis 幂等写入 (使用 SETNX)
public class IdempotentRedisSink extends RichSinkFunction<Event> {
    private transient Jedis jedis;
    
    @Override
    public void open(Configuration parameters) {
        jedis = new Jedis("localhost");
    }
    
    @Override
    public void invoke(Event event, Context context) {
        String idempotencyKey = "event:" + event.getEventId();
        
        // SET key value NX EX 3600 - 仅当key不存在时才设置，1小时过期
        String result = jedis.set(idempotencyKey, "1", "NX", "EX", 3600);
        
        if ("OK".equals(result)) {
            // 首次处理，执行写入
            processEvent(event);
        } else {
            // 重复数据，跳过
            log.info("Duplicate event skipped: {}", event.getEventId());
        }
    }
}

5.4 Savepoint 与状态升级

🎯 学习目标

  • 理解 Savepoint 与 Checkpoint 的区别

  • 学会从 Savepoint 恢复作业

5.4.1 Savepoint vs Checkpoint

Savepoint 与 Checkpoint 对比:
┌─────────────────────────────────────────────────────────────────┐
│ Checkpoint                                                      │
│ ├─ 自动周期性触发                                               │
│ ├─ 主要用于故障恢复                                             │
│ ├─ 可配置保留策略 (RETAIN/DELETE)                                │
│ └─ 存储路径通常自动管理                                          │
│                                                                 │
│ Savepoint                                                       │
│ ├─ 手动触发 (用户/运维操作)                                      │
│ ├─ 用于计划内维护、升级、迁移                                    │
│ ├─ 长期保留，不会自动删除                                        │
│ ├─ 包含作业元数据，可在不同集群恢复                              │
│ └─ 可指定自定义存储路径                                          │
└─────────────────────────────────────────────────────────────────┘

使用场景:
- Checkpoint: 日常运行中的故障自动恢复
- Savepoint: 
  ├─ 作业版本升级 (代码变更)
  ├─ Flink 版本升级
  ├─ 集群迁移
  ├─ 修改作业并行度
  └─ 数据回放和调试

// 自定义状态迁移逻辑
public class UpgradedJob extends KeyedProcessFunction<String, Event, Result> {
    
    private ValueState<NewState> newState;
    private ValueState<OldState> oldState; // 保留旧状态引用
    
    @Override
    public void open(Configuration parameters) {
        // 新状态
        newState = getRuntimeContext().getState(
            new ValueStateDescriptor<>("newState", NewState.class)
        );
        
        // 尝试读取旧状态 (如果存在)
        try {
            oldState = getRuntimeContext().getState(
                new ValueStateDescriptor<>("oldState", OldState.class)
            );
        } catch (Exception e) {
            // 旧状态不存在或类型不匹配
            oldState = null;
        }
    }
    
    @Override
    public void processElement(Event event, Context ctx, Collector<Result> out) 
            throws Exception {
        // 迁移逻辑: 如果新状态为空但有旧状态，进行迁移
        if (newState.value() == null && oldState != null && oldState.value() != null) {
            OldState old = oldState.value();
            NewState migrated = migrateOldToNew(old);
            newState.update(migrated);
            oldState.clear(); // 清理旧状态
        }
        
        // 正常处理逻辑...
    }
}
