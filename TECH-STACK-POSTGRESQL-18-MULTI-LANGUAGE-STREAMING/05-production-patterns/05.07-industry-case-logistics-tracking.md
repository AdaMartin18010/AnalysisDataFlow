## 1. 概念定义 (Definitions)

**Def-TS-26-01** (物流追踪事件流)
物流追踪事件流定义为从包裹全生命周期中采集的、带地理标签和状态标记的有序事件序列：

$$\mathcal{E}_{logistics} \triangleq \langle e_1, e_2, \ldots, e_n \rangle$$

其中每个事件 $e_i = \langle ts_i, parcel_{id}, loc_i, state_i, attrs_i \rangle$ 包含：

- $ts_i \in \mathbb{R}^+$: Unix 时间戳（毫秒精度）
- $parcel_{id} \in \mathcal{U}$: 包裹唯一标识（UUIDv7 推荐）
- $loc_i = \langle lat_i, lon_i, alt_i \rangle \in [-90,90] \times [-180,180] \times \mathbb{R}$: WGS-84 地理坐标
- $state_i \in \mathcal{S}$: 包裹状态，$\mathcal{S} = \{ \text{created}, \text{pickup}, \text{in\_transit}, \text{arrived}, \text{out\_for\_delivery}, \text{delivered}, \text{exception} \}$
- $attrs_i$: 扩展属性字典（温度、湿度、震动、承运商、车型等）

事件流满足**时间单调性**：$\forall i < j: ts_i \leq ts_j$。

**Def-TS-26-02** (包裹状态机)
包裹状态机定义为一个确定性有限状态自动机（DFA），描述包裹从创建到签收的标准生命周期：

$$\mathcal{M}_{parcel} \triangleq \langle \mathcal{S}, \mathcal{S}_0, \mathcal{F}, \Delta, \mathcal{T} \rangle$$

其中：$\mathcal{S}$ 为状态集合（同 Def-TS-26-01），$\mathcal{S}_0 = \{ \text{created} \}$ 为初始状态，$\mathcal{F} = \{ \text{delivered}, \text{exception} \}$ 为终止状态集，$\Delta \subseteq \mathcal{S} \times \mathcal{E}_{event} \times \mathcal{S}$ 为状态转移关系，$\mathcal{T}: \Delta \rightarrow \mathbb{R}^+$ 为转移时间约束函数。

核心转移规则：created $\xrightarrow{下单}$ pickup $\xrightarrow{揽收}$ in_transit $\xrightarrow{到达}$ arrived $\xrightarrow{出库}$ out_for_delivery $\xrightarrow{签收}$ delivered，任何状态均可经异常上报进入 exception。

**Def-TS-26-03** (地理围栏异常检测)
地理围栏异常检测定义为对轨迹点序列进行实时空间约束验证的判定过程：

$$\text{GeoFence}(\mathcal{E}_{logistics}, \mathcal{G}) \triangleq \{ e_i \in \mathcal{E}_{logistics} \mid \neg\text{Inside}(loc_i, g) \land g \in \mathcal{G}(parcel_{id}, ts_i) \}$$

其中 $\mathcal{G}(pid, t)$ 为随时间和包裹动态变化的围栏集合，$\text{Inside}(loc, g)$ 为点 $loc$ 位于围栏 $g$ 内部的判定谓词。冷链扩展：$\text{ColdChainAlert}(e_i) \triangleq \text{Temp}(e_i) > T_{max}(route_i) \lor \text{Temp}(e_i) < T_{min}(route_i)$。

**Def-TS-26-04** (精益物流追踪架构)
精益物流追踪架构定义为以 🌿 PG18 + RisingWave 为核心、通过标准 PostgreSQL 协议服务多语言消费者的极简流处理系统：

$$\mathcal{A}_{logistics}^{lean} \triangleq \langle PG_{18}^{tracking}, RW_{realtime}, API_{Go}, UI_{TS} \rangle$$

其中 $PG_{18}^{tracking}$ 为运行 PG18 的追踪主库，$RW_{realtime}$ 为 RisingWave 流处理节点（内嵌 CDC 消费 PG18 逻辑复制流），$API_{Go}$ 为 Go 后端（标准 `pgx` 驱动直接查询 RisingWave 物化视图），$UI_{TS}$ 为 TypeScript 前端（WebSocket 接收实时推送）。组件数 $|\mathcal{A}_{logistics}^{lean}| = 4$，对比传统 MQ 架构的 7+ 组件，故障模式从 $O(2^7)$ 降至 $O(2^4)$。

## 2. 属性推导 (Properties)

**Lemma-TS-26-01** (轨迹采样密度与路径重建精度)
设 GPS 采样间隔为 $\delta$（秒），车辆平均速度为 $v$（m/s），则相邻采样点间的最大位移为 $d_{max} = v \cdot \delta$。路径重建误差 $\epsilon$ 满足：

$$\epsilon \leq \frac{v \cdot \delta}{2}$$

*证明*: 真实路径在相邻采样点之间的最大偏离发生在两点的垂直平分线上。对于匀速直线运动的最坏情况，实际路径与线性插值的偏差上界为半位移。当 $v = 30\,\text{m/s}$（108 km/h），$\delta = 10\,\text{s}$ 时，$\epsilon \leq 150\,\text{m}$。若要求 $\epsilon \leq 50\,\text{m}$（城市配送精度），则需 $\delta \leq 3.3\,\text{s}$。∎

**Lemma-TS-26-02** (状态机流转一致性条件)
包裹状态机 $\mathcal{M}_{parcel}$ 的状态转移序列 $\langle s_0, s_1, \ldots, s_k \rangle$ 满足一致性，当且仅当：(1) **有效性**：$\forall i: (s_i, e_i, s_{i+1}) \in \Delta$；(2) **无跳跃**：不存在 $s_i \rightarrow s_{i+1}$ 使得中间状态被跳过；(3) **时效性**：$ts_{i+1} - ts_i \leq \mathcal{T}(s_i, e_i, s_{i+1})$。

*工程论证*: 条件 1 和 2 通过 PG18 的 `CHECK` 约束 + 触发器强制保证；条件 3 通过 RisingWave 窗口聚合实时检测超时转移。∎

**Prop-TS-26-01** (地理围栏检测延迟上界)
在精益架构 $\mathcal{A}_{logistics}^{lean}$ 中，地理围栏异常检测的端到端延迟 $L_{fence}$ 满足：

$$L_{fence} \leq L_{pg\_write} + L_{cdc} + L_{rw\_compute} + L_{query}$$

各分量典型值：$L_{pg\_write} \approx 5\,\text{ms}$，$L_{cdc} \in [10, 500]\,\text{ms}$，$L_{rw\_compute} \in [50, 1000]\,\text{ms}$，$L_{query} \approx 10\,\text{ms}$。因此 $L_{fence} \in [75, 1515]\,\text{ms} \ll 30\,\text{s}$（业务告警阈值）。∎

## 4. 论证过程 (Argumentation)

### 为什么物流追踪特别适合精益架构

物流追踪有三大特征使其成为 🌿 精益架构的理想用例：

**特征一：查询模式高度集中**。客户 95% 以上查询为"我的包裹现在在哪？"（单点查询，按 `parcel_id`）和"这条线路延误了吗？"（范围查询，按 `route_id + time_window`）。这两种模式均可通过 RisingWave 物化视图的 PostgreSQL 协议直接服务，无需额外的消息队列和流处理层。

**特征二：实时性要求可被物化视图满足**。客户查询延迟 < 100ms：RisingWave 物化视图查询 P99 < 50ms[^1]；异常告警 < 30秒：RisingWave CDC 延迟 + 物化视图刷新延迟 < 5秒[^2]。精益架构的延迟余量充足，无需引入 Kafka 来"削峰"。

**特征三：状态流转天然适合 SQL 表达**。包裹状态机（created → pickup → in_transit → ...）本质上是**时态表上的状态转换**。RisingWave 的 `EMIT ON WINDOW CLOSE` 语义与物流状态流转完美匹配。

### 地理时序数据的 PG18 优化策略

**第一层：存储优化 — 时序分区 + BRIN**。

```sql
CREATE TABLE gps_tracks (
    id UUIDv7 PRIMARY KEY, parcel_id UUID NOT NULL,
    ts TIMESTAMPTZ NOT NULL,
    geom GEOMETRY(POINT, 4326) NOT NULL,
    speed_kmh DECIMAL(5,2), heading SMALLINT
) PARTITION BY RANGE (ts);

CREATE TABLE gps_tracks_2026_05_06 PARTITION OF gps_tracks
    FOR VALUES FROM ('2026-05-06') TO ('2026-05-07');

CREATE INDEX idx_gps_tracks_ts_brin ON gps_tracks USING BRIN (ts) 
    WITH (pages_per_range = 128);
CREATE INDEX idx_gps_tracks_geom ON gps_tracks USING GIST (geom);
```

BRIN 索引对于只增不减的时序数据，存储开销仅为 B-Tree 的 $1/100$[^3]。

**第二层：查询优化 — GIST + KNN**。GIST 索引的 `<->` 算子支持 K-最近邻排序，可在不计算全部距离的情况下返回最近邻[^4]。

**第三层：CDC 优化 — 并行逻辑复制**。PG18 默认启用并行逻辑复制流，RisingWave 内嵌 CDC 引擎可同时消费多个复制流[^5]。

### 异常检测规则的 SQL 可表达性

| 异常类型 | 检测规则 | RisingWave SQL 表达 |
|---------|---------|-------------------|
| **延误检测** | 当前状态持续时间超过时效约束 | `TUMBLE` 窗口 + `MAX(ts) - MIN(ts) > threshold` |
| **地理围栏越界** | 轨迹点超出预设线路缓冲区 | `ST_Distance(geom, route_geom) > buffer_meters` |
| **冷链温度超标** | 温度传感器读数超出阈值窗口 | `HOPPING` 窗口 + `AVG(temp) > max_temp` |
| **丢件检测** | 包裹超过 N 小时无新事件 | `NOW() - MAX(ts) > interval '6 hours'` |
| **暴力分拣** | 加速度传感器峰值超过阈值 | `MAX(accel_g) > 5.0` 在滑动窗口内 |

这些规则在传统架构中需要 Flink CEP 或自定义流处理代码实现，而在 🌿 精益架构中仅需标准 SQL。

## 5. 形式证明 / 工程论证 (Proof / Engineering Argument)

**Thm-TS-26-01** (包裹状态机最终一致性定理)

设物流追踪系统中有 $N$ 个并发包裹的状态流转事件写入 PG18，RisingWave 通过逻辑复制消费这些变更。则在任意时刻 $t$，RisingWave 物化视图 `mv_parcel_status` 中包裹 $p$ 的状态 $s_{rw}(p, t)$ 与 PG18 中包裹 $p$ 的最新已提交状态 $s_{pg}(p, t)$ 满足最终一致性：

$$\exists \Delta_{max} > 0: \forall t: \lim_{\tau \rightarrow t + \Delta_{max}} s_{rw}(p, \tau) = s_{pg}(p, t)$$

其中 $\Delta_{max}$ 为 CDC 最大延迟（PG18 逻辑复制槽位消费滞后上限）。

*工程论证*:

1. **PG18 的 WAL 保证**: PG18 的预写日志（WAL）保证所有已提交事务的变更按提交顺序持久化[^5]。逻辑复制协议（pgoutput）按照 WAL 中的 LSN（Log Sequence Number）严格顺序发送变更事件。

2. **RisingWave CDC 语义**: RisingWave 内嵌的 Debezium Embedded Engine 消费 pgoutput 流，维护消费位点（LSN）。PG18 的 `confirmed_flush_lsn` 确保 RisingWave 确认消费后，PG 才回收 WAL 段，保证零丢失[^2]。

3. **物化视图增量维护**: RisingWave 对每个 CDC 源表维护内部状态，通过 Differential Dataflow 的增量计算保证物化视图与源表在延迟边界内一致[^1]。

4. **最终一致性收敛**: 设 PG18 的写入吞吐为 $\lambda$，逻辑复制槽的消费吞吐为 $\mu$。当 $\mu > \lambda$ 时，消费队列长度 $Q(t)$ 满足 $dQ/dt = \lambda - \mu < 0$，队列将收敛至稳态长度 $Q^* \approx 0$。因此 $s_{rw}$ 将在有限时间内追平 $s_{pg}$。∎

**Thm-TS-26-02** (地理围栏异常检测完备性定理)

设地理围栏集合为 $\mathcal{G} = \{ g_1, \ldots, g_m \}$，轨迹事件流为 $\mathcal{E}$。RisingWave 物化视图 `mv_geo_fence_breach` 基于以下 SQL 定义：

```sql
CREATE MATERIALIZED VIEW mv_geo_fence_breach AS
SELECT g.parcel_id, g.ts, g.geom, g.route_id,
    ST_Distance(g.geom::geography, r.route_geom::geography)::DECIMAL(10,2) AS dist_meters
FROM gps_tracks_stream g
JOIN route_fences r ON g.route_id = r.route_id
WHERE ST_Distance(g.geom::geography, r.route_geom::geography) > r.buffer_meters;
```

则该物化视图满足**完备性**：对于任意围栏越界事件 $e \in \mathcal{E}$，若 $e$ 触发越界条件（即 $\neg\text{Inside}(loc(e), g(e))$），则 $e$ 必将在有限延迟内出现在 `mv_geo_fence_breach` 中。

*证明*:

1. **SQL 语义等价性**: `ST_Distance(geom, route_buffer_geom) > r.buffer_meters` 与 $\neg\text{Inside}(loc, g)$ 在欧氏空间中逻辑等价（当 $g$ 定义为路径缓冲区时，距离大于缓冲区宽度等价于点在缓冲区外部）。PostGIS 的 `ST_Distance` 对 `GEOMETRY` 类型使用精确算法，对球面坐标使用 Vincenty 公式，精度满足物流场景需求[^4]。

2. **流式增量计算**: RisingWave 的物化视图维护引擎对每个新到达的 `gps_tracks_stream` 事件执行增量计算。`WHERE` 子句作为选择谓词在流处理阶段应用，只有满足条件的事件才会被物化到视图中。

3. **无遗漏保证**: PG18 CDC 保证所有写入 `gps_tracks` 表的事件被 RisingWave 消费（Thm-TS-26-01）。因此，任何触发越界条件的轨迹点都会被 RisingWave 处理，不会遗漏。∎

## 6. 实例验证 (Examples)

### 示例 1: PG18 表设计 — 包裹状态 + 轨迹时序

```sql
-- 物流追踪数据库 schema (PostgreSQL 18 + PostGIS)
CREATE EXTENSION IF NOT EXISTS postgis;
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- 包裹主表
CREATE TABLE parcels (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    tracking_no VARCHAR(32) UNIQUE NOT NULL,
    sender_info JSONB NOT NULL, receiver_info JSONB NOT NULL,
    current_state VARCHAR(32) NOT NULL CHECK (current_state IN (
        'created','pickup','in_transit','arrived',
        'out_for_delivery','delivered','exception')),
    current_location GEOMETRY(POINT, 4326),
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW(),
    expected_delivery TIMESTAMPTZ, route_id UUID,
    service_type VARCHAR(16) DEFAULT 'standard'
);
CREATE INDEX idx_parcels_location ON parcels USING GIST(current_location)
    WHERE current_state IN ('in_transit', 'out_for_delivery');

-- 状态流转事件表（事件溯源 + CDC 源）
CREATE TABLE parcel_events (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    parcel_id UUID NOT NULL REFERENCES parcels(id),
    from_state VARCHAR(32), to_state VARCHAR(32) NOT NULL,
    location GEOMETRY(POINT, 4326), facility_id UUID, operator_id UUID,
    event_data JSONB, created_at TIMESTAMPTZ DEFAULT NOW()
) PARTITION BY RANGE (created_at);
CREATE TABLE parcel_events_2026_05 PARTITION OF parcel_events
    FOR VALUES FROM ('2026-05-01') TO ('2026-06-01');
CREATE INDEX idx_parcel_events_ts_brin ON parcel_events USING BRIN (created_at);

-- GPS 轨迹表（高频写入，每日百万级）
CREATE TABLE gps_tracks (
    id UUID DEFAULT gen_random_uuid(), parcel_id UUID NOT NULL,
    device_id VARCHAR(64) NOT NULL, ts TIMESTAMPTZ NOT NULL,
    geom GEOMETRY(POINT, 4326) NOT NULL, speed_kmh DECIMAL(5,2),
    heading SMALLINT CHECK (heading >= 0 AND heading < 360),
    accuracy_meters DECIMAL(6,2), raw_data JSONB
) PARTITION BY RANGE (ts);
CREATE TABLE gps_tracks_2026_05_06 PARTITION OF gps_tracks
    FOR VALUES FROM ('2026-05-06') TO ('2026-05-07');
CREATE INDEX idx_gps_tracks_ts_brin ON gps_tracks USING BRIN (ts);
CREATE INDEX idx_gps_tracks_geom ON gps_tracks USING GIST (geom);
CREATE INDEX idx_gps_tracks_parcel ON gps_tracks(parcel_id, ts DESC);

-- 冷链温度日志表
CREATE TABLE cold_chain_logs (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    parcel_id UUID NOT NULL REFERENCES parcels(id),
    sensor_id VARCHAR(64) NOT NULL, ts TIMESTAMPTZ DEFAULT NOW(),
    temperature_c DECIMAL(4,1) NOT NULL,
    humidity_percent DECIMAL(5,2), battery_percent DECIMAL(5,2),
    alarm_flags SMALLINT DEFAULT 0
) PARTITION BY RANGE (ts);

-- 线路地理围栏表
CREATE TABLE route_fences (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    route_id UUID NOT NULL, route_name VARCHAR(128),
    route_geom GEOMETRY(LINESTRING, 4326),
    buffer_meters DECIMAL(8,2) DEFAULT 5000,
    buffer_geom GEOMETRY(POLYGON, 4326) GENERATED ALWAYS AS (
        ST_Buffer(route_geom::geography, buffer_meters)::geometry
    ) STORED, valid_from TIMESTAMPTZ, valid_to TIMESTAMPTZ
);
CREATE INDEX idx_route_fences_buffer ON route_fences USING GIST (buffer_geom);

-- 触发器：自动更新 parcels 的 current_state
CREATE OR REPLACE FUNCTION update_parcel_status() RETURNS TRIGGER AS $$
BEGIN
    UPDATE parcels SET current_state = NEW.to_state,
        current_location = NEW.location, updated_at = NOW()
    WHERE id = NEW.parcel_id;
    RETURN NEW;
END; $$ LANGUAGE plpgsql;

CREATE TRIGGER trg_parcel_event_insert
    AFTER INSERT ON parcel_events FOR EACH ROW
    EXECUTE FUNCTION update_parcel_status();
```

### 示例 2: RisingWave 实时异常检测 SQL

```sql
-- RisingWave: 物流实时分析物化视图（直接消费 PG18 CDC，零 Kafka / 零 Flink）

-- 创建 CDC 源（直连 PG18）
CREATE SOURCE parcels_source WITH (
    connector = 'postgres-cdc', hostname = 'pg18-logistics.internal',
    port = '5432', username = 'replicator', password = 'secret',
    database.name = 'logistics', slot.name = 'rw_logistics_slot'
);

-- 自动同步表
CREATE TABLE parcels FROM parcels_source TABLE 'public.parcels';
CREATE TABLE parcel_events FROM parcels_source TABLE 'public.parcel_events';
CREATE TABLE gps_tracks FROM parcels_source TABLE 'public.gps_tracks';
CREATE TABLE cold_chain_logs FROM parcels_source TABLE 'public.cold_chain_logs';

-- 包裹最新状态物化视图（客户查询 < 100ms）
CREATE MATERIALIZED VIEW mv_parcel_status AS
SELECT p.id AS parcel_id, p.tracking_no, p.current_state,
    p.current_location, p.expected_delivery, p.service_type,
    e.event_data->>'facility_name' AS last_facility_name,
    EXTRACT(EPOCH FROM (p.expected_delivery - NOW())) / 3600 AS hours_remaining
FROM parcels p
LEFT JOIN parcel_events e ON p.id = e.parcel_id
WHERE e.created_at = (SELECT MAX(created_at) FROM parcel_events WHERE parcel_id = p.id);

-- 实时延误检测（异常告警 < 30秒）
CREATE MATERIALIZED VIEW mv_delay_detection AS
SELECT p.id AS parcel_id, p.tracking_no, p.current_state,
    MAX(e.created_at) AS last_event_at,
    EXTRACT(EPOCH FROM (NOW() - MAX(e.created_at))) / 3600 AS state_hours,
    CASE 
        WHEN p.service_type = 'express' AND p.current_state = 'in_transit' 
            AND EXTRACT(EPOCH FROM (NOW() - MAX(e.created_at))) > 86400 THEN true
        WHEN p.service_type = 'standard' AND p.current_state = 'in_transit' 
            AND EXTRACT(EPOCH FROM (NOW() - MAX(e.created_at))) > 172800 THEN true
        WHEN p.current_state = 'arrived' 
            AND EXTRACT(EPOCH FROM (NOW() - MAX(e.created_at))) > 14400 THEN true
        ELSE false
    END AS is_delayed, NOW() AS detected_at
FROM parcels p
JOIN parcel_events e ON p.id = e.parcel_id
WHERE p.current_state NOT IN ('delivered', 'exception')
GROUP BY p.id, p.tracking_no, p.current_state, p.expected_delivery, p.service_type
HAVING MAX(e.created_at) < NOW() - INTERVAL '1 hour';

-- 冷链温度异常检测
CREATE MATERIALIZED VIEW mv_temp_alert AS
SELECT parcel_id, sensor_id, window_start, window_end,
    AVG(temperature_c) AS avg_temp, MAX(temperature_c) AS max_temp,
    MIN(temperature_c) AS min_temp, COUNT(*) AS sample_count,
    CASE 
        WHEN AVG(temperature_c) > 8.0 THEN 'HIGH_TEMP'
        WHEN AVG(temperature_c) < -25.0 THEN 'LOW_TEMP'
        WHEN MAX(temperature_c) - MIN(temperature_c) > 10.0 THEN 'TEMP_FLUCTUATION'
        ELSE 'NORMAL'
    END AS alert_type
FROM TUMBLE(cold_chain_logs, ts, INTERVAL '5 MINUTES')
GROUP BY parcel_id, sensor_id, window_start, window_end
HAVING AVG(temperature_c) > 8.0 OR AVG(temperature_c) < -25.0 
    OR MAX(temperature_c) - MIN(temperature_c) > 10.0;

-- 地理围栏越界检测
CREATE MATERIALIZED VIEW mv_geo_fence_breach AS
SELECT g.parcel_id, g.ts, g.geom, g.route_id, r.route_name, r.buffer_meters,
    ST_Distance(g.geom::geography, r.route_geom::geography)::DECIMAL(10,2) AS dist_meters
FROM gps_tracks g
JOIN route_fences r ON g.route_id = r.route_id
WHERE ST_Distance(g.geom::geography, r.route_geom::geography) > r.buffer_meters
  AND g.ts >= NOW() - INTERVAL '1 hour';

-- 丢件嫌疑检测（6 小时无事件更新）
CREATE MATERIALIZED VIEW mv_lost_parcel_suspect AS
SELECT p.id AS parcel_id, p.tracking_no, p.current_state, p.current_location,
    MAX(e.created_at) AS last_event_at,
    EXTRACT(EPOCH FROM (NOW() - MAX(e.created_at))) / 3600 AS hours_since_event
FROM parcels p
LEFT JOIN parcel_events e ON p.id = e.parcel_id
WHERE p.current_state NOT IN ('delivered', 'exception')
GROUP BY p.id, p.tracking_no, p.current_state, p.current_location
HAVING MAX(e.created_at) < NOW() - INTERVAL '6 hours';
```
