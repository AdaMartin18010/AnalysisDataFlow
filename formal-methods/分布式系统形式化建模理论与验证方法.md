## 第1章 概述与数学基础

### 1.1 形式化方法概述

#### 1.1.1 形式化方法的定义与价值

形式化方法是基于严格数学基础的软件开发方法，通过形式化规约语言、数学逻辑和自动推理技术来规约、设计和验证系统。
与基于测试和模拟的传统方法不同，形式化方法能够提供**数学上的正确性保证**。

在分布式系统领域，形式化方法的价值尤为突出：

| 挑战 | 形式化方法的应对 |
|------|-----------------|
| 并发执行的不确定性 | 使用状态空间探索和模型检测验证所有可能的交错 |
| 异步通信的复杂性 | 通过进程演算和消息传递语义精确建模 |
| 故障场景的多样性 | 形式化描述故障模型，验证容错机制 |
| 状态空间爆炸 | 抽象、符号化、组合推理等技术缓解 |

**形式化方法的核心价值**：

1. **早期错误检测**：在设计阶段发现缺陷，避免后期修复的高昂成本（研究表明，设计阶段修复错误的成本是编码阶段的1/10，是生产环境的1/100）
2. **精确规约**：消除自然语言描述的歧义，确保需求理解的一致性
3. **系统验证**：数学证明系统满足关键性质（安全性、活性、一致性等）
4. **文档化**：形式化规约本身就是精确的系统文档

#### 1.1.2 形式化方法的技术栈

分布式系统的形式化建模涵盖从底层到上层的完整技术栈：

```
┌─────────────────────────────────────────────────────────────┐
│                    应用层验证                                │
│  (工作流验证、流计算验证、云原生架构验证)                      │
├─────────────────────────────────────────────────────────────┤
│                    规约语言层                                │
│  (TLA+, Promela, BPMN形式化语义, 领域特定语言)                │
├─────────────────────────────────────────────────────────────┤
│                    计算模型层                                │
│  (进程演算, Petri网, 状态机, Actor模型, Dataflow)             │
├─────────────────────────────────────────────────────────────┤
│                    数学基础层                                │
│  (集合论, 数理逻辑, 代数结构, 序理论, 范畴论)                  │
└─────────────────────────────────────────────────────────────┘
```

#### 1.1.3 形式化方法的分类

形式化方法可分为三大类：

**1. 基于模型的方法（Model-Based）**

- 使用数学模型描述系统行为
- 通过模型检测验证性质
- 代表：TLA+, Promela/SPIN, UPPAAL

**2. 基于逻辑的方法（Logic-Based）**

- 使用逻辑公式表达规约和性质
- 通过定理证明验证正确性
- 代表：Coq, Isabelle/HOL, PVS

**3. 基于代数的方法（Algebra-Based）**

- 使用代数运算组合系统组件
- 通过等式推理验证等价性
- 代表：CSP, CCS, ACP

### 1.2 数学基础

#### 1.2.1 集合论与关系代数

**基本概念**

集合论是形式化方法的基石。分布式系统的状态空间可以表示为状态的集合，状态转换可以表示为状态上的关系。

**定义 1.1（关系）**：给定集合 $A$ 和 $B$，从 $A$ 到 $B$ 的关系 $R$ 是 $A \times B$ 的子集：

$$R \subseteq A \times B = \{(a, b) \mid a \in A, b \in B\}$$

**定义 1.2（关系的复合）**：给定关系 $R \subseteq A \times B$ 和 $S \subseteq B \times C$，复合关系 $R \circ S \subseteq A \times C$ 定义为：

$$R \circ S = \{(a, c) \mid \exists b \in B: (a, b) \in R \land (b, c) \in S\}$$

**定义 1.3（自反传递闭包）**：关系 $R$ 的自反传递闭包 $R^*$ 定义为：

$$R^* = \bigcup_{n \geq 0} R^n$$

其中 $R^0 = \{(a, a) \mid a \in A\}$（恒等关系），$R^{n+1} = R^n \circ R$。

**在分布式系统中的应用**

| 概念 | 集合论表示 | 应用场景 |
|------|-----------|----------|
| 进程状态 | $S$ - 状态集合 | 描述进程的所有可能状态 |
| 状态转移 | $\delta \subseteq S \times S$ | 描述状态之间的转换关系 |
| 消息传递 | $M \subseteq P \times P \times D$ | 从发送者到接收者的数据传递 |
| 因果关系 | $\prec \subseteq E \times E$ | 事件之间的happens-before关系 |
| 可见性 | $vis \subseteq E \times E$ | 一个事件对另一个事件可见 |

**关系代数的运算规则**

```
结合律:    (R ∘ S) ∘ T = R ∘ (S ∘ T)
分配律:    R ∘ (S ∪ T) = (R ∘ S) ∪ (R ∘ T)
单位元:    R ∘ I = I ∘ R = R  (I是恒等关系)
逆关系:    (R ∘ S)⁻¹ = S⁻¹ ∘ R⁻¹
```

#### 1.2.2 数理逻辑

**命题逻辑**

命题逻辑是形式化方法的基础，用于描述系统的布尔性质。

**语法**：

- 原子命题：$p, q, r, ...$
- 逻辑连接词：$\neg$（非）、$\land$（与）、$\lor$（或）、$\rightarrow$（蕴含）、$\leftrightarrow$（等价）

**语义**：

- 真值赋值函数 $v: Prop \rightarrow \{true, false\}$
- 复合公式的真值由连接词的语义规则确定

**在分布式系统中的应用**：

- 描述局部状态性质：$p \triangleq$ "进程i处于临界区"
- 表达安全性条件：$\neg(p \land q)$ - 两个进程不能同时进入临界区

**一阶逻辑**

一阶逻辑扩展了命题逻辑，增加了量词和谓词，可以表达关于数据结构和系统状态的量化性质。

**语法**：

- 项：变量、常量、函数应用
- 原子公式：$P(t_1, ..., t_n)$ - 谓词符号应用于项
- 量词：$\forall x. \phi$（全称量词）、$\exists x. \phi$（存在量词）

**在分布式系统中的应用**：

```
互斥性质: ∀i,j. (i ≠ j) → ¬(in_cs(i) ∧ in_cs(j))
活性性质: ∀i. ◇in_cs(i)  (每个进程最终能进入临界区)
一致性: ∀x. ∀v1,v2. (read(x) = v1 ∧ read(x) = v2) → v1 = v2
```

**时序逻辑**

时序逻辑是分布式系统验证中最重要的逻辑形式，用于描述系统执行路径上的性质。

**线性时序逻辑（LTL）**

LTL公式定义在无限状态序列上，基本时序算子：

| 算子 | 含义 | 形式化定义 |
|------|------|-----------|
| $X\phi$ | 下一状态 | $\sigma \models X\phi \iff \sigma^1 \models \phi$ |
| $F\phi$ | 最终 | $\sigma \models F\phi \iff \exists i \geq 0: \sigma^i \models \phi$ |
| $G\phi$ | 总是 | $\sigma \models G\phi \iff \forall i \geq 0: \sigma^i \models \phi$ |
| $\phi U \psi$ | 直到 | $\sigma \models \phi U \psi \iff \exists j \geq 0: (\sigma^j \models \psi \land \forall 0 \leq k < j: \sigma^k \models \phi)$ |

**LTL在分布式系统中的应用**：

```
安全性: G¬(critical₁ ∧ critical₂)  (互斥)
活性: G(request → F(grant))  (请求最终得到响应)
公平性: GF(scheduled)  (无限次被调度)
无饥饿: G(request → F(critical))  (请求最终进入临界区)
```

**计算树逻辑（CTL）**

CTL在分支时间上量化，可以描述系统可能的所有执行路径。

**路径量词**：

- $A\phi$：在所有路径上 $\phi$ 成立
- $E\phi$：存在一条路径使 $\phi$ 成立

**CTL公式示例**：

```
AG(p → AFq): 总是，如果p成立，则最终q会成立
EFp: 存在一条路径使得最终p成立
AGEFp: 从任何状态都可以到达满足p的状态
```

**CTL*（CTL的扩展）**

CTL*结合了LTL和CTL的表达能力，允许路径公式和状态公式的任意嵌套。

**定义 1.4（状态公式）**：
$$\phi ::= p \mid \neg\phi \mid \phi \land \phi \mid A\psi \mid E\psi$$

**定义 1.5（路径公式）**：
$$\psi ::= \phi \mid \neg\psi \mid \psi \land \psi \mid X\psi \mid \psi U \psi \mid F\psi \mid G\psi$$

#### 1.2.3 代数结构

**半群与幺半群**

**定义 1.6（半群）**：半群是一个二元组 $(S, \cdot)$，其中 $S$ 是集合，$\cdot: S \times S \rightarrow S$ 是满足结合律的二元运算：

$$\forall a, b, c \in S: (a \cdot b) \cdot c = a \cdot (b \cdot c)$$

**定义 1.7（幺半群）**：幺半群是带有单位元的半群 $(S, \cdot, e)$：

$$\forall a \in S: a \cdot e = e \cdot a = a$$

**在分布式系统中的应用**：

- 操作日志的追加操作形成幺半群
- 状态更新操作的组合

**格与偏序集**

**定义 1.8（偏序集）**：偏序集是一个二元组 $(P, \leq)$，其中 $\leq$ 是满足自反性、反对称性和传递性的二元关系。

**定义 1.9（格）**：格是一个偏序集 $(L, \leq)$，其中任意两个元素都有最小上界（join，记作 $\lor$）和最大下界（meet，记作 $\land$）。

**定义 1.10（半格）**：

- 交半格：只有meet运算
- 并半格：只有join运算

**CRDT（冲突无关复制数据类型）**

CRDT基于半格结构实现最终一致性：

```
状态合并: merge(s₁, s₂) = s₁ ∨ s₂  (并半格的join操作)
性质:
  - 幂等性: merge(s, s) = s
  - 交换性: merge(s₁, s₂) = merge(s₂, s₁)
  - 结合性: merge(merge(s₁, s₂), s₃) = merge(s₁, merge(s₂, s₃))
```

**常见的CRDT类型**：

| 类型 | 半格结构 | 应用场景 |
|------|---------|----------|
| G-Counter | 向量取最大值 | 分布式计数器 |
| PN-Counter | 两个G-Counter的差 | 支持增减的计数器 |
| G-Set | 集合的并 | 只增集合 |
| 2P-Set | G-Set + 墓碑集合 | 支持删除的集合 |
| LWW-Register | (timestamp, value)按字典序 | 最后写入获胜 |
| OR-Set | 带唯一标签的元素集合 | 有序集合 |

### 1.3 形式化规约语言

#### 1.3.1 TLA+概述

TLA+（Temporal Logic of Actions）是由Leslie Lamport开发的形式化规约语言，专门用于描述并发和分布式系统。

**核心概念**：

1. **状态（State）**：变量到值的映射
2. **动作（Action）**：状态之间的关系，表示状态转换
3. **行为（Behavior）**：无限状态序列
4. **时序公式（Temporal Formula）**：描述行为的性质

**TLA+的语法结构**：

```tla
------------------------------- MODULE ModuleName -------------------------------
EXTENDS Naturals, Sequences, FiniteSets

CONSTANTS C1, C2, ...
VARIABLES v1, v2, ...

(* 定义 *)
Def1 == ...
Def2 == ...

(* 初始状态 *)
Init == ...

(* 下一个状态关系 *)
Next == ...

(* 时序性质 *)
Spec == Init /\ [][Next]_vars /\ WF_vars(Next)

(* 不变式 *)
Invariant == ...

(* 活性性质 *)
Liveness == ...
================================================================================
```

#### 1.3.2 PlusCal算法语言

PlusCal是TLA+的算法语言，提供了更易于理解的语法糖。

**PlusCal示例 - 互斥算法**：

```pluscal
(*--algorithm Dekker
variables
  flag = [i \in {0, 1} |-> FALSE],
  turn = 0;

process Proc \in {0, 1}
variable other = 1 - self;
begin
L1: while TRUE do
L2:   flag[self] := TRUE;
L3:   while flag[other] do
L4:     if turn = other then
L5:       flag[self] := FALSE;
L6:       await turn = self;
L7:       flag[self] := TRUE;
      end if;
    end while;
CS:   critical_section;
L8:   turn := other;
L9:   flag[self] := FALSE;
    end while;
end process;
end algorithm;*)
```

**PlusCal到TLA+的转换**：

PlusCal编译器将算法转换为TLA+规范，主要包括：

1. 变量声明转换
2. 控制流转换为动作谓词
3. 标签转换为状态转换点
4. 生成不变式和活性性质的验证目标

#### 1.3.3 其他规约语言

| 语言 | 特点 | 应用场景 |
|------|------|----------|
| **Promela** | C-like语法，支持SPIN模型检测 | 通信协议验证 |
| **B Method** | 基于集合论和谓词逻辑 | 安全关键系统 |
| **Event-B** | B方法的扩展，支持逐步精化 | 系统级规约 |
| **Alloy** | 关系逻辑，基于SAT求解 | 结构建模 |
| **VDM** | 维也纳开发方法 | 软件规格说明 |

## 第2章 进程演算与计算模型

### 2.1 进程演算基础

进程演算是描述并发系统行为的形式化方法，通过代数运算组合进程，并提供等式理论推理进程等价。

#### 2.1.1 CSP通信顺序进程

**CSP（Communicating Sequential Processes）**由Tony Hoare提出，强调进程间通过消息传递进行通信。

**基本语法**：

```
P, Q ::= STOP                  (死锁)
       | SKIP                  (成功终止)
       | a → P                 (前缀)
       | P □ Q                 (外部选择)
       | P ⊓ Q                 (内部选择)
       | P ||| Q               (交错并行)
       | P |[A]| Q             (同步并行)
       | P \ A                 (隐藏)
       | P; Q                  (顺序)
       | μX.F(X)               (递归)
```

**语义模型**：

CSP有三种主要的语义模型：

| 模型 | 观察内容 | 验证能力 |
|------|---------|----------|
| **迹模型（Traces）** | 所有可能的迹 | 安全性 |
| **失败模型（Failures）** | 迹 + 拒绝集 | 安全性 + 部分活性 |
| **失败-发散模型（Failures-Divergences）** | 失败 + 发散 | 完整验证 |

**迹语义**：

进程 $P$ 的迹语义 $\mathcal{T}(P)$ 是 $P$ 可以执行的所有有限动作序列的集合。

**定义 2.1（迹）**：
$$\mathcal{T}(STOP) = \{\langle\rangle\}$$
$$\mathcal{T}(a \rightarrow P) = \{\langle\rangle\} \cup \{\langle a \rangle \frown tr \mid tr \in \mathcal{T}(P)\}$$
$$\mathcal{T}(P \square Q) = \mathcal{T}(P) \cup \mathcal{T}(Q)$$
$$\mathcal{T}(P \,\|\|\|\, Q) = \{tr \mid \exists tr_P \in \mathcal{T}(P), tr_Q \in \mathcal{T}(Q): tr \in tr_P \,\|\|\|\, tr_Q\}$$

**精化关系**：

**定义 2.2（迹精化）**：$P \sqsubseteq_T Q$ 当且仅当 $\mathcal{T}(Q) \subseteq \mathcal{T}(P)$

**定义 2.3（失败精化）**：$P \sqsubseteq_F Q$ 当且仅当 $\mathcal{F}(Q) \subseteq \mathcal{F}(P)$

**CSP应用示例 - 哲学家就餐问题**：

```csp
PHIL(i) = think →
          pickup.i.(i+1)%N →
          pickup.i.i →
          eat →
          putdown.i.i →
          putdown.i.(i+1)%N →
          PHIL(i)

FORK(i) = pickup.(i-1)%N.i → putdown.(i-1)%N.i → FORK(i)
        □ pickup.i.i → putdown.i.i → FORK(i)

SYSTEM = ||| i:0..N-1 @ (PHIL(i) ||| FORK(i))
```

**FDR模型检测器**：

FDR（Failures Divergences Refinement）是CSP的模型检测工具，可以自动验证进程之间的精化关系。


#### 2.1.2 CCS通信系统演算

**CCS（Calculus of Communicating Systems）**由Robin Milner提出，基于动作和标记转移系统。

**基本语法**：

```
P, Q ::= 0                  (空进程)
       | α.P               (前缀)
       | P + Q             (和)
       | P | Q             (并行)
       | P \ L             (限制)
       | P[f]              (重标记)
       | A                 (常量)

α ::= a                   (输入)
    | ā                   (输出)
    | τ                   (内部动作)
```

**操作语义（SOS规则）**：

$$
\frac{}{\alpha.P \xrightarrow{\alpha} P} (PREFIX)
$$

$$
\frac{P \xrightarrow{\alpha} P'}{P + Q \xrightarrow{\alpha} P'} (SUM-L)
$$

$$
\frac{Q \xrightarrow{\alpha} Q'}{P + Q \xrightarrow{\alpha} Q'} (SUM-R)
$$

$$
\frac{P \xrightarrow{\alpha} P'}{P | Q \xrightarrow{\alpha} P' | Q} (PAR-L)
$$

$$
\frac{P \xrightarrow{a} P', Q \xrightarrow{\bar{a}} Q'}{P | Q \xrightarrow{\tau} P' | Q'} (COM)
$$

**互模拟（Bisimulation）**：

**定义 2.4（强互模拟）**：关系 $\mathcal{R}$ 是强互模拟，如果 $(P, Q) \in \mathcal{R}$ 且 $P \xrightarrow{\alpha} P'$ 蕴含存在 $Q'$ 使得 $Q \xrightarrow{\alpha} Q'$ 且 $(P', Q') \in \mathcal{R}$，反之亦然。

**定义 2.5（强互模拟）**：$P \sim Q$ 当且仅当存在强互模拟 $\mathcal{R}$ 使得 $(P, Q) \in \mathcal{R}$。

**弱互模拟**：

考虑内部动作 $\tau$ 的互模拟称为弱互模拟（或观察等价）。

**定义 2.6（弱转移）**：$P \xRightarrow{\alpha} Q$ 表示 $P$ 可以通过零个或多个 $\tau$ 转移，然后执行 $\alpha$，再经过零个或多个 $\tau$ 转移到 $Q$。

**定义 2.7（弱互模拟）**：关系 $\mathcal{R}$ 是弱互模拟，如果 $(P, Q) \in \mathcal{R}$ 且 $P \xrightarrow{\alpha} P'$ 蕴含存在 $Q'$ 使得 $Q \xRightarrow{\alpha} Q'$ 且 $(P', Q') \in \mathcal{R}$，反之亦然。

**CCS应用示例 - 缓冲区**：

```ccs
B₀ = in.B₁
B₁ = in.B₂ + out.B₀
B₂ = out.B₁

-- 两个缓冲区组合
B = (B₀ | B₀[out/in]) \ {out}
```

#### 2.1.3 π演算

**π演算**是CCS的扩展，支持移动性（mobility）概念，允许通道本身作为消息传递。

**基本语法**：

```
P, Q ::= 0
       | π.P
       | P + Q
       | P | Q
       | (νx)P           (限制)
       | !P              (复制)
       | [x=y]P          (匹配)
       | [x≠y]P          (不匹配)

π ::= x(y)              (输入)
    | x̄y                (输出)
    | τ
```

**关键特性**：

1. **名称传递**：可以传递通道名称，实现动态链接
2. **范围扩展**：通过名称传递扩展限制的范围
3. **表达能力**：可以编码λ演算，是图灵完备的

**π演算语义示例**：

```
x(y).P | x̄z.Q  →  P{z/y} | Q   (通信)

(νx)(x(y).P | x̄z.Q)  →  (νx)(P{z/y} | Q)  (限制内的通信)
```

**π演算的变体**：

| 变体 | 特点 | 应用 |
|------|------|------|
| **Spi演算** | 增加加密原语 | 安全协议分析 |
| **Dpi演算** | 分布式π演算 | 分布式网络建模 |
| **Ambient演算** | 增加位置概念 | 移动计算 |
| **Join演算** | 基于连接模式 | 分布式编程语言 |

### 2.2 状态机模型

#### 2.2.1 标记转移系统（LTS）

**定义 2.8（标记转移系统）**：LTS是一个三元组 $(S, A, \rightarrow)$，其中：

- $S$ 是状态集合
- $A$ 是动作集合
- $\rightarrow \subseteq S \times A \times S$ 是转移关系

**图形表示**：

```
       a
  s₁ ───→ s₂
   │
 b │
   ↓
  s₃
```

**LTS的组合**：

**定义 2.9（并行组合）**：给定两个LTS $L_1 = (S_1, A_1, \rightarrow_1)$ 和 $L_2 = (S_2, A_2, \rightarrow_2)$，它们的并行组合 $L_1 \,\|\|\|\, L_2$ 定义为：

$$S = S_1 \times S_2$$
$$A = A_1 \cup A_2$$
$$(s_1, s_2) \xrightarrow{a} (s_1', s_2) \iff s_1 \xrightarrow{a}_1 s_1' \land a \notin A_2$$
$$(s_1, s_2) \xrightarrow{a} (s_1, s_2') \iff s_2 \xrightarrow{a}_2 s_2' \land a \notin A_1$$
$$(s_1, s_2) \xrightarrow{a} (s_1', s_2') \iff s_1 \xrightarrow{a}_1 s_1' \land s_2 \xrightarrow{a}_2 s_2' \land a \in A_1 \cap A_2$$

**状态空间爆炸问题**：

并行组合的状态空间大小是各组件状态空间的乘积：

$$|S_1 \times S_2| = |S_1| \times |S_2|$$

**缓解技术**：

| 技术 | 原理 | 效果 |
|------|------|------|
| **偏序规约（POR）** | 利用独立性减少等价状态 | 指数级减少 |
| **符号模型检测** | 使用BDD表示状态集合 | 处理大状态空间 |
| **抽象** | 隐藏无关细节 | 降维 |
| **组合推理** | 分而治之 | 避免整体构造 |

#### 2.2.2 I/O自动机

**I/O自动机**是Nancy Lynch等人提出的分布式系统建模框架。

**定义 2.10（I/O自动机）**：I/O自动机是一个五元组 $(Q, q_0, I, O, H, \delta)$，其中：

- $Q$ 是状态集合
- $q_0 \in Q$ 是初始状态
- $I$ 是输入动作集合
- $O$ 是输出动作集合
- $H$ 是内部动作集合
- $\delta \subseteq Q \times (I \cup O \cup H) \times Q$ 是转移关系

**组合规则**：

两个I/O自动机 $A_1$ 和 $A_2$ 可以组合，如果它们的输出动作集合不相交（$O_1 \cap O_2 = \emptyset$）。

组合自动机 $A = A_1 \times A_2$ 的动作集合：

- 输入：$(I_1 \cup I_2) - (O_1 \cup O_2)$
- 输出：$O_1 \cup O_2$
- 内部：$H_1 \cup H_2$

**轨迹（Trace）**：

**定义 2.11（轨迹）**：轨迹是I/O自动机执行的输入/输出动作序列，忽略内部动作。

**公平性**：

**定义 2.12（公平执行）**：执行是公平的，如果每个任务（task）在无限执行中被无限次给予执行机会。

**不变式证明**：

不变式是在所有可达状态下都成立的性质。

**证明方法**：

1. 证明初始状态满足不变式
2. 证明每个转移保持不变式

### 2.3 Actor模型

**Actor模型**是Carl Hewitt提出的并发计算模型。

**定义 2.13（Actor）**：Actor是一个计算实体，具有：

## 第3章 工作流系统的形式化建模

### 3.1 Petri网基础

**Petri网**是Carl Adam Petri提出的一种图形化和数学化的建模工具。

**定义 3.1（Petri网）**：Petri网是一个四元组 $N = (P, T, F, M_0)$，其中：

- $P$ 是库所（Place）的有限集合
- $T$ 是变迁（Transition）的有限集合，$P \cap T = \emptyset$
- $F \subseteq (P \times T) \cup (T \times P)$ 是流关系
- $M_0: P \rightarrow \mathbb{N}$ 是初始标记

**图形表示**：

```
   ●──→□──→●
  /      \
 ●        ●
  \      /
   ●──→□

● = 库所（Place）
□ = 变迁（Transition）
```

**变迁使能条件**：

**定义 3.2（使能）**：变迁 $t \in T$ 在标记 $M$ 下是使能的，记作 $M[t\rangle$，如果：

$$\forall p \in P: M(p) \geq F(p, t)$$

其中 $F(p, t) = 1$ 如果 $(p, t) \in F$，否则为0。

**变迁触发**：

**定义 3.3（触发）**：如果 $M[t\rangle$，则触发 $t$ 后得到新标记 $M'$：

$$M'(p) = M(p) - F(p, t) + F(t, p)$$

**Petri网的性质**：

| 性质 | 定义 | 验证方法 |
|------|------|----------|
| **有界性** | 每个库所的托肯数有上界 | 覆盖图分析 |
| **安全性** | 每个库所最多一个托肯 | 可达图分析 |
| **活性** | 每个变迁都可以被触发 | 陷阱/死锁分析 |
| **可达性** | 标记 $M'$ 从 $M$ 可达 | 可达图/状态方程 |
| **可逆性** | 可以回到初始标记 | 可达图分析 |

#### 3.1.1 工作流网（Workflow Net）

**定义 3.4（工作流网）**：工作流网是一个Petri网 $N = (P, T, F)$，满足：

1. 有唯一的源库所 $i$（没有输入变迁）
2. 有唯一的汇库所 $o$（没有输出变迁）
3. 每个节点都在从 $i$ 到 $o$ 的路径上

**合理性（Soundness）**：

**定义 3.5（合理性）**：工作流网是合理的，如果：

1. **可完成性**：从初始标记（$i$ 有一个托肯）可达的任何标记，都可以到达最终标记（$o$ 有一个托肯）
2. **正确完成**：当 $o$ 有托肯时，其他库所都没有托肯
3. **无死变迁**：每个变迁都可以在某些执行中被触发

**合理性验证算法**：

```python
def is_sound(workflow_net):
    # 1. 检查结构正确性
    if not has_unique_source_and_sink(workflow_net):
        return False

    # 2. 构造短接网（Short-circuited net）
    short_circuit = add_transition(workflow_net, o, i)

    # 3. 检查短接网是否活且有界
    return is_live(short_circuit) and is_bounded(short_circuit)
```

**工作流模式**：

| 模式 | Petri网表示 | 描述 |
|------|------------|------|
| **顺序** | ●→□→●→□→● | 任务按顺序执行 |
| **并行分叉** | ●→□→●→□₁→●<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;└→□₂→● | 一个任务触发多个并行任务 |
| **同步** | ●→□₁→●→□→●<br>●→□₂→●↗ | 多个任务汇合后执行 |
| **选择** | ●→□₁→●<br>&nbsp;&nbsp;↘□₂→● | 从多个分支中选择一个 |
| **循环** | ●→□₁→●→□₂→●<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;↖←←←┘ | 重复执行直到条件满足 |

#### 3.1.2 高级Petri网

**着色Petri网（CPN）**：

**定义 3.6（着色Petri网）**：CPN是一个九元组 $CPN = (\Sigma, P, T, A, N, C, G, E, I)$，其中：

- $\Sigma$ 是颜色集合（类型）
- $P, T, A$ 是库所、变迁、弧的集合
- $N: A \rightarrow P \times T \cup T \times P$ 是节点函数
- $C: P \rightarrow \Sigma$ 是颜色函数
- $G: T \rightarrow Expr$ 是守卫函数
- $E: A \rightarrow Expr$ 是弧表达式函数
- $I: P \rightarrow Expr$ 是初始化函数

**CPN示例 - 哲学家就餐**：

```cpn
-- 颜色集合
colset PHIL = index ph with 1..N;
colset FORK = index f with 1..N;

-- 库所
place Idle: PHIL;
place HasLeft: PHIL;
place Eating: PHIL;
place Forks: FORK;

-- 变迁
trans TakeLeft(ph) =
  guard: true
  input: Idle(ph), Forks(ph)
  output: HasLeft(ph)

trans TakeRight(ph) =
  guard: true
  input: HasLeft(ph), Forks((ph+1)%N)
  output: Eating(ph)

trans PutDown(ph) =
  guard: true
  input: Eating(ph)
  output: Idle(ph), Forks(ph), Forks((ph+1)%N)
```

**层次Petri网**：

层次Petri网支持模块化建模，通过子网（substitution transition）抽象复杂行为。

```
┌─────────────────────────────────────┐
│  顶层网                              │
│  ●→[子网A]→●→[子网B]→●              │
└─────────────────────────────────────┘
       ↓           ↓
┌──────────┐  ┌──────────┐
│ 子网A    │  │ 子网B    │
│ ●→□→●    │  │ ●→□→□→●  │
└──────────┘  └──────────┘
```

**时间Petri网**：

时间Petri网为变迁或库所附加时间约束。

**定义 3.7（时间Petri网）**：时间Petri网是一个Petri网，每个变迁 $t$ 有一个时间间隔 $[e, l]$，表示 $t$ 使能后必须等待至少 $e$ 时间单位才能触发，且在 $l$ 时间单位内必须触发（如果仍然使能）。

## 第4章 流计算的形式化建模

### 4.1 流计算模型

#### 4.1.1 Dataflow模型

**Google Dataflow模型**提出了流计算的核心抽象：

**核心概念**：

| 概念 | 描述 | 形式化 |
|------|------|--------|
| **事件时间** | 数据产生的时间 | $t_e$ |
| **处理时间** | 数据被处理的时间 | $t_p$ |
| **水位线** | 事件时间的进度标记 | $W(t_p)$ |
| **窗口** | 按时间边界分组数据 | $[start, end)$ |

**窗口类型**：

```
滚动窗口 (Tumbling Window):
  [0, 5) [5, 10) [10, 15) ...
  固定大小，不重叠

滑动窗口 (Sliding Window):
  [0, 5) [2, 7) [4, 9) ...
  固定大小，可重叠

会话窗口 (Session Window):
  由活动间隙定义
  动态大小
```

**水位线形式化**：

**定义 4.1（水位线）**：水位线 $W(t_p)$ 是一个单调不减函数，表示在时间 $t_p$ 时，事件时间小于 $W(t_p)$ 的所有事件都已到达。

$$\forall t_p, t_p': t_p \leq t_p' \Rightarrow W(t_p) \leq W(t_p')$$

**迟到数据处理**：

```
事件分类:
- 准时: event_time < watermark
- 迟到: event_time >= watermark

处理策略:
- 丢弃
- 允许迟到（with allowed lateness）
- 侧输出（side output）
```

#### 4.1.2 时间语义

**时间域**：

#### 4.2.2 一致性保证

**一致性级别**：

| 级别 | 保证 | 实现复杂度 |
|------|------|-----------|
| **At-most-once** | 消息最多处理一次 | 低 |
| **At-least-once** | 消息至少处理一次 | 中 |
| **Exactly-once** | 消息恰好处理一次 | 高 |

**Exactly-once实现方法**：

**1. 两阶段提交（2PC）**：

```
阶段1 (Prepare):
  - 协调者询问所有参与者是否可以提交
  - 参与者准备提交，锁定资源

阶段2 (Commit/Rollback):
  - 如果所有参与者同意，协调者发送Commit
  - 如果有参与者拒绝，协调者发送Rollback
```

**2. 分布式快照（Chandy-Lamport算法）**：

```
快照算法:
1. 快照协调器发送Barrier到所有源
2. 源在Barrier处快照状态，传递Barrier
3. 算子收到Barrier后快照状态，传递Barrier
4. 当所有Barrier到达Sink，快照完成

恢复:
- 从最近的完整快照恢复
- 重放快照后的消息
```

**Flink的Exactly-once实现**：

```
检查点机制:
- 异步快照状态
- Barrier对齐保证一致性
- 支持增量检查点

与Sink的集成:
- 幂等写入（Idempotent writes）
- 两阶段提交（Two-phase commit）
- 预写日志（Write-ahead log）
```

**控制器正确性**：

|------|------|--------|

| **活性** | 在合理时间内完成调谐 | $G(desired \neq actual \rightarrow F(desired = actual))$ |

**基于TLA+的控制器验证**：

```tla
------------------------------- MODULE KubernetesController -------------------------------
EXTENDS Naturals, Sequences, FiniteSets

CONSTANTS Resources, Controllers

VARIABLES desired, actual, controllerState

Init ==
  /\ desired = [r \in Resources |-> "default"]
  /\ actual = [r \in Resources |-> "default"]
  /\ controllerState = [c \in Controllers |-> "idle"]

Reconcile(c) ==
  /\ controllerState[c] = "idle"
  /\ LET r == CHOOSE r \in Resources : TRUE  \* 简化假设
     IN IF actual[r] # desired[r]
        THEN /\ actual' = [actual EXCEPT ![r] = desired[r]]
             /\ controllerState' = [controllerState EXCEPT ![c] = "working"]
        ELSE /\ UNCHANGED <<actual, controllerState>>
  /\ UNCHANGED desired

Complete(c) ==
  /\ controllerState[c] = "working"
  /\ controllerState' = [controllerState EXCEPT ![c] = "idle"]
  /\ UNCHANGED <<desired, actual>>

Next ==
  /\ \E c \in Controllers : Reconcile(c) \/ Complete(c)

Spec == Init /\ [][Next]_<<desired, actual, controllerState>>

(* 性质 *)
EventualConsistency ==
  \A r \in Resources : desired[r] = "target" ~> actual[r] = "target"
================================================================================
```

```dafny
// 二分查找验证
method BinarySearch(a: array<int>, key: int) returns (index: int)
  requires forall i, j :: 0 <= i < j < a.Length ==> a[i] <= a[j]  // 有序
  ensures 0 <= index ==> index < a.Length && a[index] == key
  ensures index < 0 ==> forall k :: 0 <= k < a.Length ==> a[k] != key
{
  var lo, hi := 0, a.Length;
  while lo < hi
    invariant 0 <= lo <= hi <= a.Length
    invariant forall i :: 0 <= i < lo ==> a[i] < key
    invariant forall i :: hi <= i < a.Length ==> key < a[i]
  {
    var mid := (lo + hi) / 2;
    if a[mid] < key {
      lo := mid + 1;
    } else if key < a[mid] {
      hi := mid;
    } else {
      return mid;
    }
  }
  return -1;
}
```
