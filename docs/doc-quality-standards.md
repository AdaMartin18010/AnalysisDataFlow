# 文档质量标准规范

> **所属阶段**: Knowledge | **前置依赖**: [AGENTS.md](../AGENTS.md) | **形式化等级**: L2

本文档定义了AnalysisDataFlow项目的文档质量标准，用于指导文档创建和PR审查。

---

## 1. 质量标准概述

### 1.1 质量门禁目标

- **一致性**: 确保所有文档遵循统一的结构和格式
- **完整性**: 保证文档包含必要的形式化元素和可视化内容
- **正确性**: 验证技术内容的准确性和引用的有效性
- **可维护性**: 便于后续的更新和维护

### 1.2 适用范围

本标准适用于以下目录的新建和重大修改文档：

- `Struct/` - 形式理论、分析论证、严格证明
- `Knowledge/` - 知识结构、设计模式、商业应用
- `Flink/` - Flink专项文档

---

## 2. 强制质量要求

### 2.1 六段式模板（必须）

所有核心文档必须包含以下章节：

| 序号 | 章节名称 | 英文名称 | 必需内容 |
|------|----------|----------|----------|
| 1 | 概念定义 | Definitions | 至少1个`Def-*`编号的形式化定义 |
| 2 | 属性推导 | Properties | 至少1个`Lemma-*`或`Prop-*`编号 |
| 3 | 关系建立 | Relations | 与其他概念的关联、映射、对比 |
| 4 | 论证过程 | Argumentation | 辅助定理、反例、边界讨论 |
| 5 | 形式证明 | Proof | 主要定理的完整证明或工程论证 |
| 6 | 实例验证 | Examples | 简化实例、代码片段、配置示例 |
| 7 | 可视化 | Visualizations | 至少1个Mermaid图表 |
| 8 | 引用参考 | References | 使用`[^n]`格式的引用列表 |

### 2.2 形式化元素（≥3个）

必须包含至少3个形式化元素，编号格式：

```
{类型}-{阶段}-{文档序号}-{顺序号}
```

| 类型 | 缩写 | 示例 | 用途 |
|------|------|------|------|
| 定义 | Def | `Def-S-01-01` | 核心概念定义 |
| 定理 | Thm | `Thm-K-02-01` | 主要定理陈述 |
| 引理 | Lemma | `Lemma-S-03-01` | 辅助引理 |
| 命题 | Prop | `Prop-F-04-01` | 性质命题 |
| 推论 | Cor | `Cor-S-02-01` | 定理推论 |

阶段代码：

- `S` = Struct
- `K` = Knowledge
- `F` = Flink

### 2.3 Mermaid图表（≥1个）

所有文档必须包含至少一个Mermaid图表，支持的类型：

- `graph TD/TB/LR` - 层次结构、映射关系
- `flowchart TD` - 决策树、流程图
- `stateDiagram-v2` - 状态转移、执行树
- `classDiagram` - 类型/模型结构
- `sequenceDiagram` - 时序图
- `gantt` - 路线图、时间线

### 2.4 文档头部信息（必须）

每个文档必须在标题后包含元信息：

```markdown
# 文档标题

> 所属阶段: [Struct|Knowledge|Flink] | 前置依赖: [文档链接] | 形式化等级: L[1-6]
```

形式化等级说明：

- **L1**: 概念介绍，无形式化内容
- **L2**: 半形式化描述，有数学符号但无严格证明
- **L3**: 形式化定义，有严格的数学定义
- **L4**: 形式化定理，有定理陈述但证明可略
- **L5**: 完整证明，包含详细证明过程
- **L6**: 机器验证，通过形式化工具验证

## 3. 推荐质量要求

### 3.1 代码示例

- 使用语法高亮，指定语言类型
- 代码应可运行或经过验证
- 复杂代码应附带说明注释
- 优先使用实际项目中的代码

### 3.2 引用规范

- 使用 `[^n]` 上标格式
- 在文档末尾集中列出所有引用
- 引用格式：

  ```markdown
  [^1]: 作者, "标题", 出版物, 年份. URL
  ```

- 优先引用权威来源（学术论文、官方文档、经典书籍）

### 3.3 文件名规范

- 全部小写字母
- 使用连字符 `-` 分隔单词
- 前缀体现所属层级和序号（如 `01-stream-processing-basics.md`）
- 扩展名使用 `.md`

## 4. 质量检查流程

### 4.1 本地检查

在提交PR前，使用质量门禁脚本自检：

```bash
# 检查单个文件
python .scripts/doc-quality-gate.py your-file.md --verbose

# 检查所有文件
python .scripts/doc-quality-gate.py --check-all

# JSON格式输出
python .scripts/doc-quality-gate.py your-file.md --json
```

### 4.2 CI集成检查

PR提交后会自动触发以下检查：

1. **交叉引用检查** - 验证内部链接有效性
2. **链接健康检查** - 检查所有链接状态
3. **定理编号检查** - 验证定理编号唯一性
4. **文档质量检查** - 验证六段式、形式化元素、Mermaid图表
5. **虚构API参数检查** - 检测TODO/FIXME/占位符

### 4.3 评分标准

| 检查项 | 权重 | 通过标准 |
|--------|------|----------|
| 文档头部 | 15% | 包含完整元信息 |
| 六段式模板 | 25% | 6个必需章节完整 |
| 形式化元素 | 25% | ≥3个且编号正确 |
| Mermaid图表 | 15% | ≥1个 |
| 引用格式 | 10% | 使用`[^n]`格式 |
| 代码示例 | 10% | 有代码块（可选） |

**通过标准**：总体得分 ≥ 80分，且所有强制项通过

## 5. 常见问题与解决

### 5.1 形式化元素不足

**问题**: 文档只有2个形式化元素

**解决**:

- 将重要概念提升为定义
- 从定义推导性质作为命题
- 添加辅助引理支持主要定理

### 5.2 缺少Mermaid图表

**问题**: 文档没有可视化内容

**解决**:

- 添加概念层次图展示概念关系
- 添加流程图说明算法流程
- 添加对比矩阵比较不同方案

### 5.3 引用格式错误

**问题**: 使用了 `[1]` 而非 `[^1]` 格式

**解决**:

- 全文替换为 `[^n]` 格式
- 在文档末尾添加引用定义
- 确保引用编号连续

### 5.4 章节编号混乱

**问题**: 章节编号不连续或重复

**解决**:

- 使用 `## 1. 章节名` 格式
- 子章节使用 `###` 而非编号
- 检查编号连续性

## 6. 质量门禁配置

### 6.1 脚本配置

`.scripts/doc-quality-gate.py` 支持以下配置：

```bash
# 检查指定文件
python .scripts/doc-quality-gate.py file1.md file2.md

# PR模式（只检查变更文件）
python .scripts/doc-quality-gate.py --pr-mode

# 详细输出
python .scripts/doc-quality-gate.py --verbose file.md

# JSON输出
python .scripts/doc-quality-gate.py --json file.md
```

### 6.2 CI工作流

质量门禁集成在 `.github/workflows/pr-quality-gate.yml` 中：

```yaml
- name: Document Quality Check
  run: python .scripts/doc-quality-gate.py --pr-mode --verbose
```

### 6.3 豁免规则

以下文件豁免质量检查：

- `README.md` - 项目根目录说明
- `CHANGELOG.md` - 变更日志
- `CONTRIBUTING.md` - 贡献指南
- `AGENTS.md` - Agent工作上下文
