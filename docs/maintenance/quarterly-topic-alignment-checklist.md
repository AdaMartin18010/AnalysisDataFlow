# 季度主题-子主题相关性审查清单

> **审查周期**: 每季度 (Q1/Q2/Q3/Q4) | **建议时间**: 季度最后一周
> **负责人**: 核心维护团队 + 社区代表
> **目标**: 确保 `Struct/` → `Knowledge/` → `Flink/` 三层映射的完整性与时效性

---

## 1. 审查目的

AnalysisDataFlow 项目的核心价值在于建立**形式化理论、工程知识、Flink 实践**之间的严格推导链。随着文档数量增长（900+ 篇）和外部技术生态快速演进，定期检查主题-子主题的相关性，是防止知识孤岛、保证导航有效性的关键机制。

---

## 2. 审查准备

### 2.1 前置检查清单

- [ ] 确认本季度新增/修改文档清单（通过 `git log` 或 `.scripts/stats-updater/`）
- [ ] 下载最新的 `EXTERNAL-LINK-HEALTH-REPORT` 和 `DOCUMENT-QUALITY-AUDIT`
- [ ] 收集本季度外部重大发布（Flink 新版本、新 FLIP、新协议标准、新论文）

### 2.2 工具与模板

| 工具/模板 | 用途 | 路径 |
|-----------|------|------|
| 交叉引用检查器 | 验证 Markdown 内部链接完整性 | `.scripts/cross-ref-checker-v2.py` |
| 六段式验证器 | 检查核心文档结构合规性 | `.scripts/six-section-validator.py` |
| 定理依赖分析器 | 检查形式化元素编号冲突 | `.scripts/theorem-dependency-analyzer.py` |
| 外部链接检测器 | 扫描高优先级失效链接 | `.scripts/link_checker.py` |

## 4. 评分标准

### 4.1 评分规则

- **完全符合**: 3 分
- **基本符合**: 2 分
- **部分符合**: 1 分
- **不符合**: 0 分

### 4.2 等级判定

| 总分 | 等级 | 说明 |
|------|------|------|
| 70-75 | 🟢 **优秀** | 相关性健康，只需常规维护 |
| 60-69 | 🟡 **良好** | 存在少量薄弱点，需制定改进计划 |
| 45-59 | 🟠 **警告** | 存在明显缺口，需优先修复 |
| < 45 | 🔴 **严重** | 结构完整性受威胁，需立即启动全面审查 |
